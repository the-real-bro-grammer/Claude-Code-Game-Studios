## Summary

Input System is Money Miner's touch dispatcher — the single authority that converts raw Unity touches into typed, tier-budgeted events for every consuming system. It owns the tap-vs-drag classifier, the safe-area-aware pointer state, the finger-occlusion offset that keeps tools visible above the thumb, and the input-suppression gate that prevents ghost taps during Prep→Rush phase transitions or modal dialogs. Every system that reads player intent (Tool System's drag-to-place, Collection System's tap-to-catch, Biome Map's swipe-to-pan, HUD's button taps) consumes Input's events — no system reads `UnityEngine.InputSystem` APIs directly.

> **Quick reference** — Layer: `Foundation` · Priority: `MVP` · Key deps: none (zero-dep foundation) · Publishes on Event Bus: `input.tap.main`, `input.drag.start`, `input.drag.end` (T1 Feedback), `input.gesture.longpress.unhandled` (T2 Feedback, HUD REQUIRED subscriber per R3.c + AC-R3d-required + AC-R3d-audio — dual-channel: offset visual tick above thumb + audio tick via `audio.input.longpress.received`), `input.contact.began` (T2 Feedback advisory, opt-in HUD/VFX pending-state hook per R3.e pass-4)

## Overview

Mobile touch input looks trivial until you ship it. A "tap" is not a single Unity event — it is a `began` → `moved` → `ended` sequence that must be classified as tap vs drag vs hold within a bounded time/distance window. A "drag" that starts on a tool icon must not double-fire as a "tap" on the playfield beneath. A finger hovering over the bottom third of the phone must not occlude the critical tool it is placing. A tap during a Modal/Dialog overlay must be suppressed. A player whose phone is a Samsung Galaxy S-series at 20:9 aspect ratio must see the same hit-zones as a player on a 19.5:9 iPhone. These are the problems Input System owns — and every consuming system downstream trusts Input to have already solved them before its handler runs.

The system provides four capabilities: (1) **typed event publication** — Input subscribes to `UnityEngine.InputSystem.EnhancedTouch.Touch.activeTouches` on the main thread and fires typed events (`TapPayload`, `DragPayload`) on Event Bus Feedback channels at T1 tier, plus `LongPressUnhandledPayload` on a T2 channel (opt-in); Input claims a ≤ 4 ms main-thread sub-budget inside the 16.67 ms frame (R13.c / D.1) so consumers can gate same-frame feedback while leaving ≥ 12.67 ms for render, physics, and other systems; (2) **pointer state query** — a pure `InputState.CurrentPointer` struct (position, pressed, age) that any gameplay system can read in `Update` / `FixedUpdate` without subscribing to events, for systems that need continuous state (Tool System's drag-to-place preview) rather than discrete events; (3) **phase-aware suppression** — Input queries the current Level phase (Prep / Rush / Paused) and Modal/Dialog stack depth, suppressing or routing events accordingly so ghost taps cannot fire during phase transitions, pause, or modal overlays; (4) **platform safe-area and hit-inflation contracts** — every published screen coordinate is clipped to the device safe area (notch + home indicator + punch-hole camera islands), and the 44pt iOS / 48dp Android minimum hit-target rule plus the 1.4× visual-hitbox inflation rule (per `.claude/docs/technical-preferences.md`) are enforced at the Input layer so every consumer inherits correct hitboxes without duplicate logic.

The system enforces five rules: **no direct `UnityEngine.InputSystem` access outside Input** (CI-enforced grep — consumers read only `InputEvent.*` events and `InputState.CurrentPointer`); **no input during `LoadingFromDisk`** (Save/Load holds the pause gate; Input's suppression mirrors the `SaveManager` state); **no tap synthesis from drag** (a drag that ends within the tap-classification window still fires `input.drag.end`, never `input.tap.main` — downstream systems that want either must subscribe to both); **no main-thread blocking** (Input's `Fire` → handler chain completes synchronously within Input's 4 ms main-thread sub-budget per R13.c; any consumer whose handler would overrun the 0.5 ms per-handler threshold must defer via `Task.Run` or the next frame via `EventBusPlayerLoopSystem`); **no gameplay rules in Input** (Input publishes raw pointer intent; whether a tap *hits* a coin is Collection System's concern, whether a drag *places* a ramp is Tool System's concern, whether a tap *advances* a dialog is Modal/Dialog's concern).

This GDD defines the design contract — events, states, suppression rules, tuning thresholds. Unity Input System package configuration (Input Actions asset structure vs `EnhancedTouch` direct polling; dispatch mode choice between `PlayerInput` component, generated C# class, or `InputAction` callbacks) is an implementation detail scoped for a future ADR when MVP implementation begins. Consumers designing against Input System may rely on the contract described here.

## Player Fantasy

Input System has no player-facing fantasy — it is the channel through which every other system's fantasy arrives. No player will ever say "I love this game's touch handler." They will say "that ramp snapped exactly where I wanted it" or "the gopher went for the bait the instant I tapped," and this is the system that made those sentences true.

The fantasy Input serves is Pillar 5's promise — **No Input Goes Silent**. Every tap and drag must reach its consuming system (Tool, Collection, HUD, Biome Map) within a single frame so that audiovisual feedback can fire inside the 100ms window the player's nervous system treats as "instant." Input is the first 16.67ms of that budget. Everything downstream — the tool spawn, the coin pop, the squeak — depends on this system getting out of the way fast enough that the other systems have time to be satisfying.

Picture the moment a player cares about most: Rush phase, 18 seconds on the clock, a gopher stream bending toward the left ledge, and the player drags a ramp from the tray with their right thumb. Their finger is directly over the ramp. They lift. The ramp is exactly where they meant it — offset cleanly above the point of contact, snapped to the grid, with a satisfying *clack* inside 100ms. They never thought about their finger. They thought about the gophers.

That is the fantasy Input protects — and it is not Input's fantasy. It belongs to Tool System, to Collection, to the Prep→Rush handoff. Input's job is to be the channel that delivers the tap and drag so faithfully that the player never notices Input exists. The tap-vs-drag classifier, the 1.4× hitbox inflation, the spawn-above-finger offset, the suppression gate that kills stale taps crossing the Prep→Rush boundary — every one of these exists to keep the player's attention on the board and off their hand.

The fantasy is NOT "my tap was registered." The fantasy is "I never had to wonder whether my tap was registered." Silence when working, invisible when working. When Input is doing its job, the player is thinking about gophers, not fingers.

## Detailed Design

### Core Rules

**R1. Single input authority.** No code outside `InputDispatcher.cs` and `InputState.cs` accesses `UnityEngine.InputSystem.*` namespaces (including `EnhancedTouch`, `LowLevel`). Consumers read only `InputEvent.*` typed events on Event Bus and the `InputState.CurrentPointer` struct. Enforcement — CI grep pattern `^using UnityEngine\.InputSystem(\.EnhancedTouch|\.LowLevel)?;` applied to all `.cs` files with allowlist: `src/Input/InputDispatcher.cs`, `src/Input/InputState.cs`, `tests/unit/Input/**/*.cs`, `tests/integration/Input/**/*.cs`. Any match outside the allowlist fails CI.

**R2. First-touch only at MVP.** Input tracks the single active touch identified by the finger ID of the first touch to enter `began` phase. Any simultaneous second-or-higher-finger contact is silently ignored (no event published, no log). The tracked finger is released on `ended` or `canceled`. Multi-touch (pinch-to-zoom on Biome Map, dual-thumb gestures) is explicitly post-MVP and requires an ADR to repeal R2. Enforcement — PlayMode test: simulated two-finger simultaneous `began` produces exactly one event sequence (from the first finger only).

**R3. Gesture classification.**

- **R3.a Tap.** Movement delta `< 10pt` (iOS) / `< 10dp` (Android) for the full touch duration AND total duration `≤ 300ms`. Published as `input.tap.main` on the `ended` phase — **never on `began`**. A tap-that-becomes-a-drag generates zero taps. Late-lift forgiveness: a touch lifting at 301ms+ with `< 10pt` movement and no velocity is still classified as a tap (motor-impaired accommodation).
- **R3.b Drag.** Movement delta `≥ 10pt` / `10dp` at any time during the touch, OR delta `> 6pt` / `6dp` after 150ms (slow-deliberate-drag early trigger, critical for Rush-phase ramp placement where waiting 300ms for classification wastes 18% of the 100ms feedback window). On the frame the threshold is crossed, `input.drag.start` fires with the original `began` position as `StartScreen` and the current position as `EndScreen`. On `ended`, `input.drag.end` fires with the original `began` position and the final position.
- **R3.c Long-press — reserved at MVP, published as unhandled with REQUIRED MVP dual-channel subscriber.** Duration `> t_tap_max` with `d < d_slop` and lift-velocity `v ≥ v_lateLift` is classified as LongPress. Input publishes `input.gesture.longpress.unhandled` on a T2 Feedback channel (not T1) with payload `LongPressUnhandledPayload(ScreenPosition, DurationMs, Target?)`. **HUD is a REQUIRED MVP subscriber** — HUD registers on scene start and, on every LongPress classification, renders a **dual-channel feedback contract** (pass-3 BLOCKING #1 resolution — `ScreenPosition` under-the-thumb occlusion failure):
    1. **Visual tick (occlusion-offset)** — sprite rendered at `ScreenPosition + InputState.GetOcclusionOffset(ScreenPosition)` (i.e. `+72 dp` upward clamped to `safeArea.top − 8 dp` per R7), so the feedback appears above the thumb-pad that just long-pressed (not under the finger still in contact). Minimum visual spec handed to HUD GDD: sprite diameter ≥ 48 dp (matches the accessibility hit-target floor), duration ≈ 200 ms dim/fade, **minimum 3:1 contrast against the expected HUD/biome background** (art-director to define the canonical sprite in HUD GDD / asset spec; default shipping treatment is a flat neutral-grey disc with thick outline consistent with the Cartoon Heist Diorama art direction).
    2. **Audio tick (Pillar 5 non-visual fail-safe)** — a short audio cue published on Event Bus Feedback channel `audio.input.longpress.received` (4-segment slug conforming to Event Bus R5 regex `^(audio|vfx)\.[a-z_]+\.[a-z_]+\.[a-z_]+$` — namespace corrected in pass-4 BLOCKING #11 from the pass-3 draft `input.longpress.received`, which would have namespace-collided with Input's own `input.*` Feedback channels). HUD subscribes to this channel and routes it through Audio Bus for playback; the canonical audio asset tag is owned by the Audio Bus GDD (pending authoring). The audio tick is timed to the same 200 ms window as the visual tick. The audio channel is the guaranteed non-visual feedback path — it works when the visual is occluded by hands, bright sunlight, or low-vision conditions, covering the exact accessibility population the 600 ms `t_tap_max` ceiling exists to serve.
  This dual-channel contract is the minimum needed to honour Pillar 5's "no input goes silent" promise; a single visual-only tick at raw `ScreenPosition` (the pass-2 spec) fails the perceptual contract for motor-impaired players still in contact at lift, and is superseded by this spec (see review log 2026-04-22 pass-3 BLOCKING #1). At VS, Accessibility Service may extend or replace either channel with a richer treatment; both HUD channels remain the MVP fail-safe. Post-MVP systems that require long-press semantics can subscribe alongside HUD without repealing R3.c. `input.tap.main`, `input.drag.start`, and `input.drag.end` never fire for a LongPress-classified gesture. Enforcement — CI asserts ≥1 T2 subscriber is registered at scene start on any scene that spawns `InputDispatcher` (see AC-R3d-required). The HUD GDD is the owner of the dual-channel implementation spec; this GDD defines the contract, not the sprite asset or audio-tag filename.
- **R3.d Continuous drag state via polling, not events.** No `input.drag.update` event exists. Systems needing per-frame drag position (Tool System's drag-to-place preview) read `InputState.CurrentPointer` in `Update`. Rationale: per-frame bus traffic at 60fps would blow T1 subscriber-handler budgets; polling is cheaper and self-throttles to each consumer's own frame cadence.

- **R3.e Contact-Began advisory event (pass-4 BLOCKING #13 — honours pass-1 CD adjudication condition).** On the `began` phase of every tracked touch (subject to R10 suppression), Input publishes `input.contact.began` on a T2 Feedback channel with payload `ContactBeganPayload(Vector2 ScreenPosition, int FingerID, float BeganRealtimeSeconds)`. This is an **advisory event** — no REQUIRED MVP subscriber; HUD and VFX/Juice MAY subscribe for on-contact pending-state feedback (a minimal "contact received" visual or audio tick that extinguishes on the subsequent `input.tap.main` / `input.drag.start` / `input.gesture.longpress.unhandled` for the same FingerID). Rationale: R3.a requires tap to fire on `ended` (preserves tap-vs-drag disambiguation per D.3) but that produces up to 300 ms / 600 ms of on-contact silence before any T1 event reaches subscribers — a structural Pillar 5 regression for the slow-tap / accommodation subpopulation during Rush phase. CD's pass-1 adjudication of game-designer's BLOCKING finding was *conditional* on exposing this T2 advisory; three passes deferred the condition. Pass-4 honours it. The classifier's Tap / Drag / LongPress outputs are unchanged — R3.e fires IN ADDITION to whatever the classifier eventually resolves. Subscribers MUST correlate `ContactBeganPayload.FingerID` with the subsequent event's FingerID to extinguish their pending-state UI (the FingerID is added to `TapPayload` / `DragPayload` / `LongPressUnhandledPayload` if not already present — flag for consistency-check). If R10 suppression is active at `began`, `input.contact.began` is NOT published (mirrors AC-INV2 invariant — no events in Suppressed state, extended to T2 `contact.began` channel). Per-frame budget impact: one additional T2 publish per gesture lifetime (at `began`) — negligible against the T2 100 ms Pillar 5 envelope and not counted against Input's 4 ms main-thread sub-budget (T2 dispatch is asynchronous from R13.c's main-thread measurement — confirm at Event Bus consistency check). **Event Bus impact**: one new row in Event Bus Interactions table (Emitter: Input System, Event: `input.contact.began`, Namespace: Feedback, Payload: `ContactBeganPayload`, Subscribers: HUD opt-in, VFX/Juice opt-in VS, Latency: T2) — pending Event Bus GDD amendment at Input pass-4 merge.

Enforcement — unit tests for each branch (at-threshold, one unit below, one unit above); integration test confirming no `input.tap.main` fires after a drag; AC-R3g asserts `input.contact.began` fires on every tracked `began` (suppression-aware).

**R4. Hit-target resolution order.** For `TapPayload.Target`, Input performs a two-pass resolution on the main thread:

1. **UI raycast** — Clear the pre-allocated `List<RaycastResult>`, populate `PointerEventData` with the touch's safe-area-local screen position, and call `EventSystem.current.RaycastAll(pointerEventData, results)` (caller-allocated list, zero-alloc on the Input side). If `results.Count > 0` and `results[0].gameObject != null`, treat the first result as `Target` and SKIP Pass 2. **Presence of raycast results is the authoritative UI-hit signal.** Do NOT call `EventSystem.IsPointerOverGameObject(int touchId)`: when `InputSystemUIInputModule` is the active UI input module (standard under Unity 6.3's new Input System), the `touchId` from `EnhancedTouch.Touch.touchId` does not map to the EventSystem's internal pointer IDs — the module synthesises its own pointer IDs and does not pass raw OS touch IDs through. Using the overload produces unreliable results. **UNVERIFIED — engineering task during implementation**: confirm that `PanelRaycaster` (UI Toolkit) participates in `EventSystem.RaycastAll` results in Unity 6.3 LTS; if it does not, UI Toolkit panels would fall through to Pass 2 (world raycast) and the R4 "UI first" ordering would break for any UI Toolkit content. Add a PlayMode test with a UI Toolkit panel over a world collider on first implementation and promote this caveat to a resolution before merge.
2. **World raycast** — `Physics2D.OverlapPointNonAlloc(worldPoint, resultsBuffer, layerMask)` with pre-allocated 8-element buffer and layer-filtered collision matrix. First hit wins. `Physics2D.OverlapPoint` (allocating) is BANNED — CI grep enforces this.

If both passes return null, `Target = null` (playfield tap). Rationale — reversing the order would let a tap pass through a modal overlay and reach the playfield beneath (ghost tap bug). ScreenSpaceOverlay canvases are outside the camera frustum and cannot be detected by Physics2D queries; EventSystem raycast is the only correct UI path. Enforcement — PlayMode test: UI layer over a world collider; published `Target` is the UI object.

**R5. Coordinate contract — safe-area-local pixels.**

- **R5.a Safe-area clipping.** Touches whose `began` position lies outside `Screen.safeArea` (iOS notch, home indicator, Android punch-hole exclusion zones, back-gesture edge) are suppressed at source — no event published, no `CurrentPointer` update. Publishing-and-filtering at the consumer layer creates a race where fast taps reach T2 subscribers before their own filter runs.
- **R5.b Every published `Vector2` is safe-area-local.** Origin is `Screen.safeArea.bottomLeft`. Consumers do NOT add their own safe-area offset. The safe-area rect is cached once on scene load (it does not change during a session on any supported iOS/Android device). A touch that begins inside safe area and moves outside remains tracked; positions are clamped to the safe-area boundary on each `Update`.

Enforcement — unit test with mock safe-area rect: touch position outside → no event; edge-crossing touch → coordinates clamped.

**R6. Hitbox inflation is consumer-authored, not Input-enforced.** Input publishes raw finger contact coordinates. Consuming systems (Tool System, Collection System, HUD, Biome Map, Modal/Dialog) are responsible for authoring their colliders and UI RectTransforms with the 1.4× visual-sprite-bounds inflation and the 44pt iOS / 48dp Android minimum hit-target floor (per `.claude/docs/technical-preferences.md`). Rationale — a centralized `InflatedBoundsRegistry` at the Input layer would require every dynamic spawn/despawn (gophers mid-Rush, consumables mid-deploy) to register/deregister with Input, impacting the 60fps frame budget and adding state coupling that Unity's native collider model does not require.

- **R6.a `DeviceDisplayMetrics` singleton.** Input exposes a read-only `DeviceDisplayMetrics` struct (pixels-per-point, pixels-per-dp, safe-area rect, screen aspect ratio) populated in `Initialize()` before any touch event fires. Consumers reference this to derive correct sizes without platform `#ifdef`s. Registered as a cross-system constant in `design/registry/entities.yaml` (post-authoring Section 5b).
- **R6.b Debug-build floor assertion.** In `UNITY_EDITOR` and `DEVELOPMENT_BUILD`, every consuming system's UI element / collider registered through the generic `IInteractable.RegisterHitBounds(Rect)` helper asserts that the bounds width AND height meet the accessibility floor. Violations produce a blocking assertion (not a silent clamp — silent clamps hide authoring errors). Release builds omit the assertion.

**R7. Finger-occlusion offset for spawn-above-finger.** When a consuming system (Tool System) requests a spawn position for an object dragged under the thumb, Input exposes `InputState.GetOcclusionOffset(Vector2 touchPoint) → Vector2`. Default offset: `+72dp` on the Y axis (≈1.5× a 48dp average thumb-pad diameter). If `touchPoint.y + 72dp > Screen.safeArea.top`, clamp to `Screen.safeArea.top - 8dp` (breathing room from ceiling). Fixed-offset over screen-relative — at 19.5:9 vs 21:9, screen-relative offsets feel inconsistent. Tuning range in Section G.

**R8. Input produces zero feedback.** No sounds, visuals, animations, haptics, or particle spawns originate from `InputDispatcher`. Input publishes a typed event on Event Bus and returns. All feedback is the subscribing systems' concern (HUD for button flash, VFX/Juice for tap pop, Audio Bus for tap SFX, Haptics for rumble). Enforcement — design-review gate: `InputDispatcher.cs` imports zero audio/visual/haptics namespaces; no `Play*()`, `Instantiate()`, `UnityEngine.Rendering.*`, `Handheld\.Vibrate`, `AndroidJavaObject`, `AndroidJavaClass`, `CoreHaptics`, `VibrationEffect`, or `\.Haptics\b` calls. (The prior grep `UnityEngine.InputSystem.XInput.Haptics.*` was replaced per review log 2026-04-22 BLOCKING #5 — that pattern is gamepad-only and never matches on touch-only mobile builds. The new pattern covers `Handheld.Vibrate` Unity built-in, iOS CoreHaptics, Android JNI vibration paths via `AndroidJavaObject` / `AndroidJavaClass`, and generic `.Haptics` namespace fragments. Over-inclusive by design; false positives are acceptable in `InputDispatcher.cs` since any JNI usage there is itself suspect.)

**R9. MVP accessibility floors — configurable, UI deferred to VS.** Input reads an `AccessibilitySettings` struct from `PlayerPrefs` on `Initialize()` with three configurable values:

| Setting | MVP Range | Default | VS UI |
|---|---|---|---|
| Tap duration ceiling (R3.a) | 300–600ms | 300ms | Yes — slider in Accessibility menu |
| Drag-distance slop (R3.b) | 10–20pt/dp | 10pt/dp | Yes — slider |
| Long-press threshold (R3.c) | 300–1000ms | 300ms (suppressed at MVP) | Deferred — feature gated |

These paths exist at MVP so the VS Accessibility Service adds UI configuration without re-architecting Input. **MVP debug-menu requirement (pass-3 BLOCKING #7 — OQ-6, + pass-4 BLOCKING #12 activation-pattern resolution)**: because `PlayerPrefs` is not player-accessible on mobile, MVP ships a **non-impaired-activation debug menu** — specifically:

1. **`DEVELOPMENT_BUILD`-gated on-screen button** — a labelled "Accessibility Debug" button is rendered in the pause-menu overlay in any `DEVELOPMENT_BUILD` build. The button is activated by a single tap; no long-press, no multi-touch. This is the PRIMARY access path for motor-impaired playtesters during MVP playtest sessions (all MVP playtest builds are Development builds).
2. **Rapid 5-tap on title-screen logo** — as a secondary access path in both Dev and shipping MVP builds, 5 taps within 2 seconds on the title-screen logo opens the same accessibility settings overlay. Rapid tap-counting does not require sustained hold and is achievable by motor-impaired users at baseline `t_tap_max = 300 ms` because each tap is independent. This path ALSO works in shipping MVP builds (not Dev-gated) so MVP public playtest users have a zero-impaired-gesture accommodation path until the v1.0 Accessibility Settings UI lands.
3. **Explicit non-requirement**: the activation MUST NOT require long-press (would require the impaired gesture to access long-press accommodation — pass-3's catch-22 flagged by ux-designer pass-4 BLOCKING #12) and MUST NOT require multi-touch (banned at MVP per R2).

Pass-3's "long-press + three-finger-tap unlock" activation suggestion is explicitly repealed — it was self-defeating for the population the accommodation serves. Both activation paths expose sliders for `t_tap_max`, `d_slop`, and long-press threshold. The full player-facing UI is a v1.0-cert-gating ADR owned by Main Menu / Settings / Pause + Accessibility Service GDDs; that ADR MUST land before shipping cert. The MVP debug menu is a temporary scaffold, not a ship-quality player UI.

**R10. Suppression gate — no events in any Suppressed state.** When `InputDispatcher.State` is `Uninitialized`, `Disposed`, or any `Suppressed.*`, all touches are discarded: no `input.*` events fire, `InputState.CurrentPointer.IsPressed == false`, `InputState.CurrentPointer.ScreenPosition` returns its last valid value (NOT zero — avoids consumers snapping to origin). Any in-flight gesture (begun but not classified) at suppression entry is silently abandoned; its eventual `ended` fires no event.

- **R10.a Phase suppression.** Input subscribes to `gameplay.level.phase_changed` on Event Bus (Gameplay namespace, T2 tier). On any `Prep→Rush`, `Rush→Paused`, or `Rush→Complete` transition, Input enters `Suppressed.PhaseTransition` for `PhaseTransitionSuppressDuration` ms (Section G knob). Input does NOT query Level Runtime directly — Level Runtime is a downstream consumer of Input; subscribing to the bus preserves zero-upstream-dependency.
- **R10.b Modal suppression.** Modal/Dialog calls `InputDispatcher.PushModalSuppression(modalId)` and `.PopModalSuppression(modalId)` directly (not via Event Bus — modal open/close must guarantee suppression before the next frame's touch processing). Depth-counted — nested modals do not prematurely lift suppression.
- **R10.c Load suppression.** Save/Load calls `InputDispatcher.SetLoadSuppression(bool)` at `LoadingFromDisk` enter/exit. Mirrors Save/Load's state machine (see save-load.md Section C, States and Transitions).
- **R10.d Priority.** `Suppressed.Loading` > `Suppressed.Modal` > `Suppressed.PhaseTransition`. All three are independent flags; `Active` is entered only when all three are simultaneously inactive. Debug-log state name is the highest-priority active flag.

**R11. Pause gesture cancellation (show-stopper fix).** On `OnApplicationPause(true)`: Input immediately abandons any in-flight gesture state, sets `_activeTouchId = -1`, and transitions to whatever suppression state is appropriate (Loading if Save/Load signals it, else a temporary resume-guard state). No `input.drag.end` fires on pause even if a drag was active — the pause itself is the cancel signal. On `OnApplicationPause(false)`: Input resets `_activeTouchId = -1` before the first post-resume `began` is processed. This prevents the Unity Input System's post-resume synthetic `ended` event (the OS reports finger-up state on resume; Unity fires `ended` against the pre-pause touch state) from firing a stale `drag.end` on a gesture the player already abandoned. Enforcement — PlayMode integration test: begin drag → trigger `OnApplicationPause(true)` → `OnApplicationPause(false)` → verify no `drag.end` fires; verify new touch after resume fires cleanly.

**R12. No gameplay logic in Input.** `InputDispatcher.cs` and `InputState.cs` do not `using` any of `MoneyMiner.Gameplay`, `MoneyMiner.Tools`, `MoneyMiner.Collection`, `MoneyMiner.Economy`, `MoneyMiner.Progression`. Input does not inspect hit-target components to decide event payload content (e.g., Input does NOT check "is this a coin?" to route to a different event). Whether a tap hits a coin, a ramp, a gopher, or a menu button is entirely the subscriber's concern. Enforcement — CI grep on `using` directives.

**R13. Input Pipeline Budget.** Input's dispatch runs on the main thread inside a single 16.67 ms frame and MUST consume only a fraction of that budget so the frame can also render, step physics, and run gameplay. Input's main-thread sub-budget is layered:

- **R13.a Pre-Fire pipeline ≤ 2 ms (Release) / ≤ 2.5 ms (Dev).** From `EnhancedTouch.Touch.activeTouches` read through tap/drag classifier through `EventSystem.RaycastAll` + `Physics2D.OverlapPointNonAlloc` through `TapPayload` / `DragPayload` construction to the `EventBus.Fire(channel, payload)` entry. Measured inside `InputDispatcher.Update()`.
- **R13.b T1 subscriber chain ≤ 2 ms (Release) / ≤ 2.5 ms (Dev).** From `Fire` entry through last subscriber return on any `input.*` channel. This is Input's *slice* of Event Bus's T1 dispatch envelope — the registry `t1_latency_max = 16.67 ms` (event-bus.md D.1) is the Event Bus's *per-channel* T1 window; Input claims 2 ms of it so the remaining ≥14 ms is available to other T1 channels firing in the same frame. **Cross-GDD aggregate coordination (pass-2 BLOCKING #4 resolution):** Input's 4 ms main-thread sub-budget (R13.c) is further constrained by Event Bus's `T1_per_frame_aggregate_ms = 8 ms` locked constant (event-bus.md D.1, pass-4 cross-coordination amendment 2026-04-22) — a *per-frame sum across all T1 channels combined*. Input's 4 ms claim is 50% of that 8 ms aggregate; remaining ≤ 4 ms is shared among all other MVP T1 producers (Collection, Tool, Hazard, Critter AI). The aggregate is enforced by Event Bus AC-F4-aggregate; any cross-producer breach is an Event Bus CI failure that attributes cost per channel, not an Input-only failure.
- **R13.c Input main-thread total ≤ 4 ms (Release) / ≤ 5 ms (Dev).** R13.a + R13.b. Leaves ≥12.67 ms of the 60 fps frame for render, physics, gameplay updates, and other T1 channels firing. **This is a sub-budget *inside the frame*, NOT the frame budget itself** — the 16.67 ms registry `t1_latency_max` constant is a *different* constraint (Event Bus per-channel latency window); AC-P1's end-to-end Pillar 5 Feedback Floor is a *third, separate* clock (100 ms at 60 fps target / 133 ms at 30 fps floor — touch-began → render-frame-submission wall-clock). All three must hold independently.
- **R13.d T1 subscriber cap + per-handler cost (independent caps).** No `input.*` channel may have more than **4** T1 subscribers simultaneously registered in a given scene (SubscriberCountMax = 4, locked). Per-handler cost threshold: **1.0 ms Release / 1.5 ms Dev** on the ADR-003 reference device (relaxed from 0.8 ms / 1.2 ms per review log 2026-04-22 pass-3 BLOCKING #8 — the pass-2 adjudication's 4:1 SNR claim at 0.8 ms was mean-vs-mean; `Stopwatch.GetTimestamp` 99p noise *tail* on Mali-G52-class ARM cores is 0.4–0.6 ms — not the 0.2 ms mean — producing 99p-tail SNR ≤ 2:1 and residual CI flakiness at the tail. The 1.0 ms Release threshold restores ≥ 2:1 tail SNR against a 0.5 ms 99p noise floor and covers compliant handlers up to ~0.4 ms actual cost; Dev 1.5 ms absorbs the additional 0.4–0.8 ms overhead that the Dev profiler instrumentation adds on top of the noise floor). **Release measurement uses inline `ProfilerMarker.Begin()` / `.End()` pairs around each subscriber invocation; per-handler samples are then read externally via `ProfilerRecorder.StartNew(ProfilerCategory.Scripts, "Input.Handler.{TypeName}.{MethodName}", 1).** Pass-3 BLOCKING #3's initial framing claimed "no inline brackets in Release" — that claim was mechanically incoherent (pass-4 BLOCKING #5 correction): `ProfilerRecorder` reads named markers, so the markers must still wrap each subscriber invocation inline to produce per-handler granularity. What pass-3 correctly achieved is migration **away from `Stopwatch.GetTimestamp` (heavier ~0.1–0.2 ms per bracket, 4 subs × 2 brackets ≈ 0.4–0.8 ms overhead)** and **toward `ProfilerMarker.Begin/End` (lighter ~0.05–0.1 ms per bracket, 4 subs × 2 brackets ≈ 0.2–0.4 ms overhead)**. The marker overhead is absorbed by the 1.0 ms Release per-handler ceiling (budget accounting: 1.0 ms ceiling = ~0.3 ms marker overhead + ~0.5 ms measurement noise floor + ≤ ~0.2 ms real handler cost on compliant subscribers; regressions above ~0.4 ms real cost inflate against the ceiling). Dev-tier AC-R13f retains inline `Stopwatch.GetTimestamp` delta brackets since Dev's wider 1.5 ms ceiling absorbs the heavier Stopwatch overhead and Dev builds are not shipping cert targets. The two measurement paths are mutually exclusive per build variant (`MONEYMINER_RELEASE_GATE` defined → Release path compiles in; absent → Dev path compiles in) so Release and Dev captures are separate recordings — see H.4 preamble tier-split note. **These are independent caps, not a multiplicative invariant.** The chain ceiling (T1Chain ≤ 2.0 ms Release / ≤ 2.5 ms Dev per D.1) is the authoritative budget constraint; the per-handler cap is a per-individual fairness rule that catches single-handler regressions the chain average would hide. The naive product `SubscriberCountMax × PerHandlerCostThreshold_ms` (4 × 1.0 = 4.0 ms) exceeds the 2.0 ms chain ceiling by design — this is the expected geometry when per-handler is a per-individual cap and chain is a joint cap: not every subscriber sits at the per-handler ceiling, and the chain cap enforces the actual frame cost. Any subscriber whose handler measures > 1.0 ms on the reference device (Release) MUST defer heavy work to `Task.Run` (off-main) or to the next frame via `EventBusPlayerLoopSystem` deferred queue. The chain cap is enforced separately by AC-R13b; the per-handler cap is enforced by AC-R13f. Both must pass independently. **Post-MVP refinement path (OQ-7):** noise-subtracted per-handler measurement (AC asserts against delta between subscriber 99p and empty-handler baseline 99p) would allow tightening the threshold back toward 0.8 ms by removing the noise floor from the comparison; deferred as a post-MVP architectural upgrade rather than an MVP gate.
- **R13.e CI gate tiers.** Development Build (with profiler markers enabled) asserts 99p `TotalLatency_ms ≤ 5.0 ms` (R13.a + R13.b); Release Build on ADR-003 reference device asserts 99p `TotalLatency_ms ≤ 4.0 ms`. The 1.0 ms Dev margin absorbs `Stopwatch.GetTimestamp` profiler-marker overhead (~0.1 ms × multiple markers) and IL2CPP stripping differences between Dev (None) and Release (High).

Enforcement — CI PlayMode instrumentation on `InputDispatcher.Update()` measures R13.a wall-clock; Event Bus AC-F4-equivalent measures R13.b; both reported per-run.

## Formulas

Input System is a dispatch layer with minimal continuous mathematics. Most threshold values (safe-area clipping, suppression durations, occlusion constants) are tuning knobs defined in Section G and referenced by rule in Section C. Two calculations require formal treatment because they have non-trivial boundary conditions or produce outputs that downstream CI tests key on.

### D.1 InputPipelineBudget

**Purpose.** Formalizes the latency constraint from R13 so that CI assertions and per-handler budget decisions have a single symbolic source to reference.

**Expression.**

```
TotalLatency_ms = PreFire_ms + T1Chain_ms

where:
  PreFire_ms   ≤ BudgetCeiling_ms / 2
  T1Chain_ms   ≤ BudgetCeiling_ms / 2
  TotalLatency_ms ≤ BudgetCeiling_ms

  BudgetCeiling_ms = 4.0    if DeviceProfile = Release   (Input's main-thread sub-budget inside a 16.67 ms frame)
  BudgetCeiling_ms = 5.0    if DeviceProfile = Dev       (absorbs profiler-marker overhead)
```

**Variable table.**

| Symbol | Type | Range | Description |
|--------|------|-------|-------------|
| `PreFire_ms` | float | 0 – 2.0 (Release) / 0 – 2.5 (Dev) | Wall-clock ms from `EnhancedTouch.activeTouches` read through `TapPayload` / `DragPayload` construction to `EventBus.Fire()` entry. Measured inside `InputDispatcher.Update()`. |
| `T1Chain_ms` | float | 0 – 2.0 (Release) / 0 – 2.5 (Dev) | Wall-clock ms from `EventBus.Fire()` entry through last T1 subscriber return on any `input.*` channel. Each subscriber's contribution is additive and serial. |
| `BudgetCeiling_ms` | float | {4.0, 5.0} | Input's main-thread sub-budget inside a 16.67 ms frame — **NOT** the registry `t1_latency_max` (which is Event Bus's per-channel T1 window, a different clock) and **NOT** AC-P1's end-to-end Pillar 5 clock (100 ms at 60 fps target / 133 ms at 30 fps floor — touch-began → render-frame-submission). 4.0 = Input's Release claim on the frame (leaves ≥12.67 ms for render + physics + other T1 channels). 5.0 = Development Build allowance for profiler-marker overhead (~0.1 ms × markers + IL2CPP-None vs IL2CPP-High stripping differences). |
| `DeviceProfile` | enum | {Release, Dev} | Build profile at CI time. Release = ADR-003 reference device with IL2CPP High stripping. Dev = editor or development build with profiler instrumentation enabled. |
| `TotalLatency_ms` | float | 0 – 5.0 (bounded by ceiling) | Sum of PreFire and T1Chain. Must not exceed BudgetCeiling for the active profile. Output is clamped by CI assertion — an overrun fails the test, it does not silently continue. |

**Output range.** Bounded above by `BudgetCeiling_ms` (4.0 Release / 5.0 Dev); bounded below by 0 (theoretical; minimum measured overhead is ~0.4 ms for empty subscriber chain). The ceiling is a CI hard gate, not a runtime clamp.

**Worked example — nominal Release build, 3 subscribers on `input.tap.main`.**

```
PreFire_ms   = 1.1   (classifier + two raycasts + payload alloc)
T1Chain_ms   = 0.9   (HUD 0.3ms + Collection 0.4ms + Modal/Dialog 0.2ms)
TotalLatency = 1.1 + 0.9 = 2.0 ms   →   2.0 ≤ 4.0   PASS

DeviceProfile = Dev, same chain:
BudgetCeiling = 5.0 ms
2.0 ≤ 5.0   PASS (comfortable margin)

(Tool System does not count here — it polls `InputState.CurrentPointer`
 in Update per R3.d rather than subscribing to input.tap.main.)
```

**Rationale.** The half-and-half split (PreFire ≤ half, T1Chain ≤ half) is a **design-intent load-balancing contract**, not a merge-blocking gate — clarified per pass-4 BLOCKING #8. It prevents one side from crowding out the other at the design level: a slow raycast pass cannot steal budget from subscribers, and a slow subscriber cannot justify loosening the pre-fire cost. The asymmetric Dev ceiling absorbs Stopwatch overhead without changing the Release target — the production experience is unchanged.

**Gate asymmetry note (pass-4 BLOCKING #8 clarification)**: only the **joint sum** (AC-R13c Release 99p-of-per-frame-sum, 4.0 ms ceiling) is a BLOCKING CI merge gate. The half-and-half per-component caps (AC-R13a PreFire 99p ≤ 2.0 ms; AC-R13b T1Chain 99p ≤ 2.0 ms) are classified PLAYMODE — they produce attribution data for failing joint-sum runs but do NOT independently block merge. This is deliberate: a build with `(PreFire = 2.5, T1Chain = 1.4, joint = 3.9)` passes AC-R13c (3.9 ≤ 4.0) and merges, accepting the sub-budget imbalance as a real-but-tolerable deviation from the design-intent split. A build with `(PreFire = 2.5, T1Chain = 2.0, joint = 4.5)` fails AC-R13c and is merge-blocked (or correlated-tail-downgraded per AC-R13c's pass-4 routing clause). Designers reading D.1 should expect the half-and-half split to be violated occasionally under thermal pressure and read AC-R13a/b failures as attribution signals for AC-R13c triage, not as independent merge blockers.

The 4.0 ms Release ceiling is explicitly a *sub-budget* inside one 16.67 ms frame — prior revisions of this GDD conflated Input's main-thread cost with the full frame budget (same number, three different clocks); that conflation is resolved here per the 2026-04-22 review.

**Validation.** CI PlayMode test on ADR-003 reference device: 1000 synthetic touch events across 60 frames; 99th-percentile `TotalLatency_ms ≤ 4.0`; mean `PreFire_ms < 1.5` and mean `T1Chain_ms < 1.5`. Dev build gate: 99th-percentile `TotalLatency_ms ≤ 5.0`.

### D.3 GestureClassification

**Purpose.** Defines the piecewise mapping from raw touch input `(d, t)` to gesture class. Replaces an implicit decision tree with explicit, gap-free boundary conditions so implementers and reviewers can verify completeness and testers can construct boundary-value cases mechanically.

**Expression.**

Let `d` = cumulative movement distance in platform-independent units (pt on iOS, dp on Android), and `t` = touch duration in milliseconds from `began` phase to current frame. `v` = finger lift velocity at `ended` (magnitude of last-frame delta / last-frame delta-time).

```
GestureClass(d, t, ended, v) =

  InProgress   if  t ≤ t_tap_max  AND  d < d_slop  AND  NOT ended
  InProgress   if  t ≤ t_early    AND  d < d_early AND  NOT ended

  Tap          if  ended  AND  d < d_slop  AND  t ≤ t_tap_max
  Tap          if  ended  AND  d < d_slop  AND  t > t_tap_max  AND  v < v_lateLift
               (late-lift forgiveness — motor-impaired accommodation, R3.a)

  Drag         if  d ≥ d_slop              (threshold crossing triggers immediately, any t)
  Drag         if  d ≥ d_early  AND  t ≥ t_early

  LongPress    if  ended  AND  d < d_slop  AND  t > t_tap_max  AND  v ≥ v_lateLift
               (published as input.gesture.longpress.unhandled on T2 per R3.c;
                HUD REQUIRED MVP subscriber per AC-R3d-required, dual-channel:
                offset visual tick above thumb + audio tick via Audio Bus)
```

**Boundary semantics.** At `d == d_slop` the Drag branch fires — the `≥` comparator is inclusive; the Tap branch's `d < d_slop` is strict. Similarly, `v == v_lateLift` at `ended` classifies as LongPress (not Tap) because the Tap late-lift branch is `v < v_lateLift` strictly. The `t ≤ t_tap_max` boundary is inclusive — a touch ending at exactly `t == t_tap_max` with `d < d_slop` classifies as Tap (standard branch, not late-lift). These boundary rules are load-bearing for H.2 Formula Correctness ACs — implementers and QA should read the comparators exactly as written.

**Statefulness.** The classifier is **stateful per-gesture**. Once a gesture publishes `input.drag.start` (the Drag branch fires on `d ≥ d_slop` OR on the early-drag gate `d ≥ d_early AND t ≥ t_early`), the gesture is locked into Drag for the remainder of its lifetime — subsequent frames cannot re-classify it as Tap or LongPress even if `d` somehow decreases or other conditions would otherwise match. The Drag lock persists until `ended` or `canceled`, at which point the classifier publishes `input.drag.end` and releases. This prevents pathological re-classification under finger-jitter or floating-point noise (a player whose drag slightly retracts mid-gesture does not see their drag disappear and become a tap). Implementers MUST maintain an explicit `_activeClassification` field on the gesture tracker and short-circuit the classifier whenever it holds `Dragging`.

**Classifier-knob margin invariant.** The OnValidate rule on the classifier ScriptableObject MUST enforce `d_early ≤ d_slop − 3` (a 3-unit margin, in pt/dp). The prior rule `d_early < d_slop` permitted settings like `(d_slop = 20, d_early = 19)` which produce a classifier **dead zone** at `d = 15, t = 160 ms, ended = false`: the touch satisfies neither Tap rules (`d < d_slop` requires `ended`, which is false), nor the early-drag gate (`d = 15 < d_early = 19`), nor the immediate-drag gate (`d = 15 < d_slop = 20`), so it remains `InProgress` indefinitely until lift — burning the 100 ms Pillar 5 response window at the accessibility ceiling tuning. The 3-unit margin ensures `d_early ≤ 17` when `d_slop = 20`, so a deliberate slow drag at `d = 15` still classifies as `InProgress` approaching the `d_early` gate, and a drag at `d = 17` classifies as Drag via the early-drag branch at `t ≥ t_early`. Reviewers should reject any PR raising `d_slop` without confirming the margin holds. See AC-D3-margin for the validation path.

**Variable table.**

| Symbol | Type | Range | Description |
|--------|------|-------|-------------|
| `d` | float | 0 – unbounded (pt/dp) | Cumulative movement distance since `began`. Computed as sum of per-frame deltas, not straight-line origin distance (handles circular motion correctly). |
| `t` | float | 0 – unbounded (ms) | Duration since the touch entered `began` phase on the current classifier frame. |
| `d_slop` | float | 10 – 20 (pt/dp) | Tap movement tolerance. Default 10. Accessible range 10–20 (Section G). Below 10: unintended drag promotions from normal motor jitter. Above 20: intended drags misclassified as taps on short swipes. |
| `d_early` | float | 6 – (d_slop − 3) (pt/dp) | Early-drag trigger distance used only when `t ≥ t_early`. Default 6. Invariant: `d_early ≤ d_slop − 3` enforced by `OnValidate` (3-unit margin prevents a classifier dead zone under max-accessibility tuning — see D.3 Classifier-knob margin invariant). |
| `t_tap_max` | int | 300 – 600 (ms) | Tap duration ceiling. Default 300 ms. Accessible range 300–600 ms (Section G). |
| `t_early` | int | 100 – 200 (ms) | Early-drag trigger time. Default 150 ms. At `t ≥ t_early`, the smaller `d_early` threshold activates to catch deliberate slow Rush-phase drags before the full tap window closes. |
| `v` | float | 0 – unbounded (pt/ms) | Finger lift velocity at `ended` phase. Used only for late-lift forgiveness branch. |
| `v_lateLift` | float | 0 – 0.1 (pt/ms) | Velocity ceiling for late-lift Tap forgiveness. Default 0.02 pt/ms. Touches at or above this velocity at lift (with `t > t_tap_max` and `d < d_slop`) classify as LongPress and publish `input.gesture.longpress.unhandled` on T2 (HUD REQUIRED MVP subscriber dual-channel); they are NOT discarded. Touches below this velocity fall through to Tap late-lift forgiveness. |
| `GestureClass` | enum | {InProgress, Tap, Drag, LongPress} | Output classification. `InProgress` is transient (never published). `Tap` → `input.tap.main` (T1). `Drag` → `input.drag.start` on threshold crossing (T1). `LongPress` → `input.gesture.longpress.unhandled` (T2 Feedback; **HUD is REQUIRED MVP subscriber per R3.c + AC-R3d-required**, dual-channel: offset visual + audio cue; post-MVP systems may subscribe alongside HUD). |

**Output range.** Categorical, four values. Every `(d, t, ended, v)` combination maps to exactly one class — the piecewise definition is exhaustive and mutually exclusive. `InProgress` is never published to Event Bus; it is an internal state. The late-lift forgiveness branch and the `LongPress` branch are the only cases that require the `ended` signal to resolve — all `Drag` promotions fire on the frame the threshold is crossed, not on lift.

**Worked examples — three paths.**

```
Path A — clean tap:
  d=3pt, t=180ms, ended=true, v=0.01 pt/ms
  d < d_slop(10), t ≤ t_tap_max(300), v < v_lateLift(0.02)   →   Tap
  Publish input.tap.main

Path B — slow deliberate drag (Rush-phase ramp placement):
  d=7pt, t=160ms, ended=false
  d ≥ d_early(6) AND t ≥ t_early(150)   →   Drag
  Publish input.drag.start this frame
  (without early trigger: d=7 < d_slop(10); touch would remain InProgress until t=300ms — 140ms late)

Path C — long press (motor-impaired, slow lift):
  d=2pt, t=420ms, ended=true, v=0.05 pt/ms
  d < d_slop, t > t_tap_max(300), v ≥ v_lateLift(0.02)   →   LongPress
  Publish input.gesture.longpress.unhandled on T2 Feedback channel
  (HUD REQUIRED MVP subscriber per R3.c renders dual-channel feedback:
   offset visual tick above thumb + audio tick via Audio Bus)
```

**Rationale.** The early-drag trigger (Path B) is the load-bearing case for Rush-phase usability: without it, a deliberate 150 ms slow drag sits in `InProgress` until the 300 ms tap window closes, wasting half of the Tool System's 100 ms response window. The two-variable threshold `(d_early, t_early)` lets tuning tighten or loosen the slow-drag sensitivity without touching the standard slop, keeping tap and drag classifier sensitivities independent. The piecewise form makes boundary tests mechanically derivable — every variable has an exact threshold, so QA writes `(threshold − 1, at-threshold, threshold + 1)` cases without guessing.

**Validation.** Unit tests covering all eight boundary conditions: `d = d_slop − 1`, `d = d_slop`, `d = d_slop + 1`, `t = t_tap_max`, `t = t_early − 1`, `t = t_early` (with `d = d_early`), late-lift forgiveness path, long-press discard path. Integration test: simulated slow Rush-phase drag (`d = 7pt` at `t = 160ms`) confirms `input.drag.start` fires at frame ≈ 160 ms, not frame 300 ms.

### Formula Candidates Moved to Tuning Knobs

The following geometric and capacity constants are tuning knobs (Section G), not formulas: `occlusion_offset_dp` (default 72) and `edge_padding_dp` (default 8) from R7's spawn-above-finger clamp `min(touch.y + offset, safeArea.top − padding)`; `PhaseTransitionSuppressDuration_ms` from R10.a; `SubscriberCountMax` (4) and `PerHandlerCostThreshold_ms` (1.0 Release / 1.5 Dev per pass-3 retune, independent of SubscriberCountMax per R13.d) from R13.d. Each is a constant with a tuning range — no interacting variables, no emergent output. Treating them as formulas would be padding.

## Edge Cases

### OS / Platform

- **If `_activeTouchId` is tracked but the ID vanishes from `EnhancedTouch.Touch.activeTouches` without firing `ended` or `canceled`** (OS-level touch steal — Android OEM gesture navigation, notification tray swipe, Samsung edge panel): `InputDispatcher.Update()` performs an absence-detection pass — if the tracked ID is not present in `activeTouches` AND no `ended`/`canceled` was received, treat as an implicit cancel: clear `_activeTouchId`, abandon the in-flight gesture, publish nothing. Log `[InputDispatcher] Tracked touch {id} vanished — implicit cancel`. This is independent of R11's pause cancellation, which only covers `OnApplicationPause`.

- **If `ended` or `canceled` arrives for a finger ID that does not match `_activeTouchId`** (spurious OS ghost event; also the aftermath of an implicit cancel): discard silently. No event published, no state change, no log in release builds. Rationale — these events are noise at the current abstraction; matching IDs is the only authoritative signal that a tracked gesture ended.

- **If `Screen.safeArea` changes mid-session** (Android foldable unfold, split-screen activation): out of MVP scope. Money Miner is portrait-locked per `technical-preferences.md` and foldables are not a supported form factor for v1.0. If a foldable unfolds during play, the cached safe-area rect becomes stale and edge-adjacent touches may misbehave until scene reload. Post-launch consideration: add a `Screen.safeArea` change check in `Update` and re-cache + emit a `DeviceDisplayMetrics` changed event.

- **If the OS recycles a touch ID after a tracked finger vanished implicitly** (EC-OS1 fired, new touch arrives with the same finger ID the OS just recycled): a `began` matching `_activeTouchId` while `IsPressed` is already true (compound failure: EC-OS1's absence-detection did not run yet) is treated as an implicit cancel of the prior gesture before starting fresh — NOT silently skipped. Without this rule, the prior gesture's state would leak into the new touch. Enforcement — PlayMode test simulating OS ID recycling.

### State Machine / Lifecycle

- **If `SetLoadSuppression(true)` fires mid-drag**: R10.c abandons the in-flight gesture and clears `_activeTouchId`. When load suppression lifts and the Unity Input System delivers the physical `ended` event for the touch that was still down during load, the arriving `ended`'s finger ID does NOT match `_activeTouchId` (cleared during abandonment). The ghost `ended` is discarded under the EC-OS2 rule — no event published. This is why EC-OS2's ID-mismatch rule is load-bearing for suppression recovery.

- **If `PushModalSuppression(modalId)` is called during a T1 subscriber handler that was triggered by Input's own `Fire`** (Modal/Dialog opens in response to a tap, calling into Input's suppression API while Input is still inside its dispatch stack): `PushModalSuppression` sets the suppression flag atomically but gesture abandonment is DEFERRED to the next `InputDispatcher.Update()` entry. Abandonment code does NOT run from inside `PushModalSuppression` — re-entering the state machine from an arbitrary subscriber call stack depth would corrupt state.

- **If `InputDispatcher.Initialize()` is called before `EnhancedTouchSupport.Enable()` completes, or called a second time without an intervening `Dispose()`** (scene reload, additive scene load): `Initialize()` asserts `State == Uninitialized` at entry and throws `InvalidOperationException` with message `[InputDispatcher] Already initialized — call Dispose() before re-initializing`. Double-registration of the `gameplay.level.phase_changed` subscription would cause every phase change to trigger two suppression entries. Enforcement — integration test.

- **If a T1 subscriber mutates Input's suppression state inside its own handler** (e.g., Tool System places a tool → fires `gameplay.level.quota_progress` → Level Runtime emits `gameplay.level.phase_changed` → Input's subscription wants to enter `Suppressed.PhaseTransition`): Input snapshots suppression flags at `Update()` entry and uses the snapshot for the entire frame's processing. Suppression mutations during the T1 chain take effect the FOLLOWING frame, not mid-dispatch. Prevents re-entrancy hazards in the dispatcher.

### Classifier Boundary

- **If a frame drops to ≥ 200 ms (severe thermal throttle or background work stall)**: D.3 uses `t = elapsed since began`, sampled per `Update`. If `began` arrives at frame N, frame N takes 220 ms, finger lifts mid-frame — `t` at `ended` is 220 ms ≤ `t_tap_max(300)`, fires as Tap correctly. **Dangerous case**: if `began` is processed in frame N (220 ms duration) and finger lifts at the START of frame N+1 (also 220 ms), `t` at `ended` = 440 ms with near-zero `v` → classified as LongPress. Per revised R3.c, this publishes `input.gesture.longpress.unhandled` on T2 (not silently discarded); the player's intended quick tap still does not reach Tap subscribers, but the T2 event exists so opt-in feedback consumers can surface "hold received" UI. A player-intended quick Tap-subscription semantic is still lost to frame-rate collapse. Accepted for MVP — Pillar 5 assumes a healthy 60 fps / 30 fps floor; sustained sub-5 fps is a thermal emergency, not a classification concern. Flag in Section I (Open Questions) if post-MVP telemetry shows late-lift taps disproportionately lost on specific devices.

- **If a second finger's `began` and the first finger's `ended` arrive in the same `Update`** (rapid multi-touch): Input processes the tracked finger's lifecycle event FIRST (publish tap / end drag, clear `_activeTouchId`), THEN evaluates any new `began`. If the new `began` is from a second finger, it is ignored per R2 (first-touch only). Processing in the wrong order would leave the first finger's gesture dangling and the second finger's `began` silently consumed. Enforcement — unit test with synthetic activeTouches buffer containing both phases.

- **If `t_tap_max` is raised to the accessibility ceiling (600 ms)** and a player holds a stationary finger for 580 ms with a slow lift (`v < v_lateLift`): classified as Tap per D.3 late-lift forgiveness. **UX discontinuity to document for playtesters**: under the 600 ms setting, extended stationary holds the player intended as "pause to think" fire as taps. This is a cost of motor-impaired accommodation. Flag in Section I for VS Accessibility Service UX review — the VS settings UI should describe this trade-off in the accessibility menu copy.

### Hit-Test Resolution

- **If the hit-tested GameObject (UI or world) is destroyed between the raycast and `EventBus.Fire`**: after `OverlapPointNonAlloc` or `RaycastAll`, Input performs a null-and-destroyed check (`target == null || !target`) before populating `TapPayload.Target`. If destroyed, `Target = null` (treat as playfield tap). Failing this check would propagate a phantom-object reference to up to 4 T1 subscribers (R13.d cap), causing cascading `MissingReferenceException`. Enforcement — PlayMode test destroying the target object synchronously inside an earlier handler.

- **If multiple UI canvases share the same `sortingOrder` at the tap point** (authoring error): `EventSystem.RaycastAll` returns results in sort-then-distance order; at identical sort order the result order is Unity-implementation-defined. A `DEVELOPMENT_BUILD` scene-validator asserts every active canvas in the scene has a unique sort order. Runtime recovery is NOT attempted — Input cannot guess authoring intent. Release builds accept whatever Unity returns; the scene validator in Dev catches this before release.

- **If the tap falls on a world collider whose layer is excluded from Input's layer-filtered mask**: `OverlapPointNonAlloc` returns zero hits; `Target = null`; the player perceives the tap as "missing" a visible object. This is intentional — Input's layer mask defines which world objects are tap-eligible, and excluded objects are visual-only per design (Data Registry + Tool System decide which ToolConfigs register on tap-eligible layers). If a consuming system needs expanded hit eligibility, it updates its collider's layer, not Input's mask. Documentation handoff to consumer GDDs.

- **If a touch begins at exactly the `Screen.safeArea` boundary pixel**: inclusive of the safe area. The rule is `touch.position IN safeArea` (closed interval on all four edges). Only touches strictly OUTSIDE the safe-area rectangle are suppressed. Edge-of-screen taps are a common player affordance (reaching a button with the edge of the thumb) and must not be lost to a strict-inequality bug.

### Configuration / Tuning Extremes

- **If a consuming system queries `InputState.DeviceDisplayMetrics` before `InputDispatcher.Initialize()` completes** (Unity script execution order violation): the query throws `InvalidOperationException` with message `[InputDispatcher] DeviceDisplayMetrics not populated — InputDispatcher must execute before consumers in Script Execution Order`. Silent return of a default value would mask authoring errors in the scene setup. `InputDispatcher` is pinned to a negative Script Execution Order number (e.g., -1000) so it always initializes before default-order MonoBehaviours. Enforcement — integration test that instantiates a consumer MonoBehaviour at default order and asserts the InvalidOperationException fires if Input is mispinned.

### Cross-System Ownership Rule

**Input System owns:**
- Touch capture and the sole `UnityEngine.InputSystem.*` access boundary (R1)
- Gesture classification thresholds — tap, drag, long-press boundary rules (R3, formula D.3)
- Safe-area coordinate contract — all published positions in safe-area-local pixels (R5)
- Finger-occlusion offset constants (R7)
- `DeviceDisplayMetrics` read-only struct (R6.a)
- Suppression gate — state, priority, and transition (R10)
- Pause-gesture cancellation (R11)
- Input pipeline budget enforcement (R13, formula D.1)

**Consumers own:**
- **Hit-target authoring** — collider and RectTransform bounds, 1.4× visual-sprite inflation, 44pt iOS / 48dp Android accessibility floor (R6 — Input does NOT enforce; R6.b debug-build floor assertion is a safety net, not a replacement for consumer discipline)
- **Feedback emission** — every sound, visual, animation, haptic, and particle is subscriber-side (R8 — Input produces zero)
- **Gameplay interpretation** — whether a tap hits a coin, a drag places a ramp, a swipe advances a map (R12 — Input does NOT route by gameplay class)
- **Per-handler budget** — each T1 subscriber keeps work ≤ 2ms on reference device or defers via `Task.Run` / next-frame `EventBusPlayerLoopSystem` (R13.d)
- **Multi-touch semantics (post-MVP)** — if a consumer needs pinch-zoom or dual-thumb gestures, it repeals R2 via ADR; Input does not silently widen its contract

**Three scope guardrails:**
- **Settings UI is not Input's.** If a value is player-facing (keybinding, tap-sensitivity slider), it belongs in Accessibility Service. Input reads accessibility state via `PlayerPrefs` (R9); Input does not own the settings screen.
- **Gameplay rules are not Input's.** Hit priority, target eligibility, and collision-layer assignment belong in the consumer (Tool, Collection, Modal). Input reports contact; consumers decide meaning.
- **Platform differences are not consumers'.** iOS vs Android differences (pt vs dp, safe-area shape, notch geometry) live inside `DeviceDisplayMetrics` (R6.a). Consumers do not `#ifdef` — they query the struct.

If these guardrails are violated, the consumer is encoding Input's contract in two places and one will eventually drift.

## Tuning Knobs

Input System's tuning knobs are a mix of **perceptual thresholds** (gesture classifier — affects feel and accessibility), **geometric constants** (finger-occlusion offset, safe-area padding — affects ergonomic correctness on each aspect ratio), **pipeline capacity** (subscriber cap, per-handler threshold — affects the 16.67 ms T1 budget), and **CI severities** (R1/R12 grep gates — affects architectural discipline). Gameplay tuning (hit priority, collision layers, target eligibility) lives in the consumer GDDs per R12, not here.

The R9 `AccessibilitySettings` struct's three values (`t_tap_max`, `d_slop`, long-press threshold) are exposed at MVP via `PlayerPrefs` with VS Accessibility Service UI as their designer-facing surface. They are listed below alongside the non-accessibility knobs because they share the same tuning range and validation discipline; at VS, the slider UI reads/writes the same keys.

### Gesture Classifier Knobs (formula D.3)

| Parameter | Current Value | Safe Range | Effect of Increase | Effect of Decrease |
|---|---|---|---|---|
| `d_slop` (R3.a, D.3) | 10 pt / dp | [10, 20] | Taps absorbed on slight intended drags (accessibility win for motor jitter); intended short swipes misclassify as taps | Unintended drag promotions from normal motor jitter; player-perceived "I tapped but the ramp started dragging" bugs |
| `d_early` (R3.b, D.3) | 6 pt / dp | [6, d_slop − 3] | Slow deliberate drags take longer to trigger — closer to 300 ms wait before `input.drag.start`; wastes Rush-phase response window | Early-trigger false positives from finger tremor; ramp may drag before player intended to commit |
| `t_tap_max` (R3.a, R9, D.3) | 300 ms | [300, 600] | Accessibility win for slow lifts (motor-impaired); UX discontinuity at 600 ms — stationary "pause to think" holds fire as taps (EC Classifier-3) | Below 300 breaks late-lift forgiveness for normal players; not permitted |
| `t_early` (R3.b, D.3) | 150 ms | [100, 200] | Slower slow-drag detection; delays Rush-phase ramp response | Earlier trigger on stationary fingers that haven't yet committed motion; false-positive drags |
| `v_lateLift` (R3.a, D.3) | 0.02 pt/ms | [0.01, 0.1] | Late-lift forgiveness covers more cases — even quick lifts after long holds classify as taps; dilutes the long-press reserve | Stricter — more late-lift touches fall through to LongPress discard (R3.c silent drop); players with slow deliberate lifts lose taps |
| Long-press threshold (R3.c, R9) | 300 ms (suppressed at MVP) | [300, 1000] | Reserved for post-MVP; unlocking is an ADR action, not a tuning knob change | N/A at MVP |

**Coupling invariant**: `d_early ≤ d_slop − 3` (3-unit margin) is enforced by `OnValidate` on the classifier config ScriptableObject. Attempting `d_early = d_slop − 3` is the tightest legal setting and IS allowed — but any configuration with `d_early > d_slop − 3` is rejected with a compile-time asset validation error. The 3-unit margin prevents a classifier **dead zone** at max-accessibility tuning (e.g. `d_slop = 20, d_early = 19` under the prior `d_early < d_slop` invariant produced an `InProgress`-indefinitely failure at `d = 15, t = 160 ms, ended = false` — the touch matched neither the `d < d_slop` Tap branch (requires `ended`), nor the `d ≥ d_early` early-drag gate, nor the `d ≥ d_slop` immediate-drag gate; see D.3 Classifier-knob margin invariant + review log 2026-04-22 BLOCKING #6). Changing `d_slop` without inspecting `d_early` is the most common tuning mistake — reviewers should flag any PR that touches one without the other.

Final value is pending two inputs: (1) the Rush-phase end-card animation duration from Level Runtime GDD (cite the animation's fade-out completion frame as the suppression floor); and (2) the `/prototype prep-react-core-loop` validation criterion — **"a Continue-button tap fired within 100 ms of end-card appearance must land"**. If the prototype observes the 150 ms default forfeiting those rapid post-transition taps, tune toward the 100 ms range; if ghost taps appear during the transition animation itself, tune toward 200–250 ms. Playtest rather than speculate.
| `Physics2D.OverlapPointNonAlloc` buffer size (R4) | 8 colliders | [4, 16] | More simultaneous hit candidates at a point (stacked UI + world); zero-alloc property preserved; larger buffer = marginal heap footprint | Truncated results on densely-layered regions; the first-hit-wins rule may miss the intended target if too many colliders overlap at the tap point |

### Interaction Between Tuning Knobs

- **Coupled — `d_slop` and `d_early`**: the formula D.3 requires `d_early ≤ d_slop − 3` (3-unit margin; see OnValidate rule and D.3 Classifier-knob margin invariant). Tuning `d_slop` upward (accessibility) without also tuning `d_early` produces a larger dead zone where slow drags are neither caught by the early trigger nor by the standard threshold. Change both together, or the slow-drag-to-ramp path (Path B in D.3 worked examples) regresses — the margin is the minimum guard against this; aggressive accessibility profiles may want a wider gap.
- **Coupled — `t_tap_max` and `v_lateLift`**: at the accessibility ceiling (`t_tap_max = 600 ms`), `v_lateLift` becomes the sole discriminator between late-lift Tap and LongPress for slow finger lifts. Raising `v_lateLift` too aggressively at 600 ms converts nearly every slow lift into a tap, eliminating the LongPress reserve that post-MVP systems may want to claim. Review together when adjusting accessibility profile.
- **Independent caps — `SubscriberCountMax` and `PerHandlerCostThreshold_ms`**: prior revisions treated these as multiplicatively coupled (`N × per_handler ≤ chain` at `4 × 0.5 = 2.0 ms`, binding, zero slack). Reversed per review log 2026-04-22 pass-2 BLOCKING #3 — the 0.5 ms per-handler threshold sat below `Stopwatch.GetTimestamp` mobile noise (~0.2 ms mean → 2.5:1 SNR, 5–8% false-positive CI runs). Pass-2 relaxation raised per-handler Release to 0.8 ms / Dev to 1.2 ms on a mean-based 4:1 SNR claim. **Pass-3 BLOCKING #8 retune**: the mean-based SNR claim was wrong at the tail — Mali-G52-class 99p Stopwatch noise is ~0.4–0.6 ms, so 0.8 ms tail SNR was still ≤ 2:1. Pass-3 restores a ≥ 2:1 tail SNR by raising per-handler Release to **1.0 ms** and Dev to **1.5 ms**, with Release measurement migrated from inline `Stopwatch.GetTimestamp` brackets (~0.1–0.2 ms per bracket, 0.4–0.8 ms per chain) to inline `ProfilerMarker.Begin/End` brackets (~0.05–0.1 ms per bracket, 0.2–0.4 ms per chain) + external `ProfilerRecorder` sample reads (pass-4 BLOCKING #5 correction — Release retains inline brackets but they are lighter ProfilerMarker brackets, not Stopwatch; `ProfilerRecorder` alone cannot produce per-handler granularity without markers to read). Under the current values, `4 × 1.0 = 4.0 ms` per-handler × subscriber count mathematically exceeds the 2.0 ms chain ceiling — that is the expected geometry when the two caps are independent: the chain ceiling (AC-R13b) is the authoritative joint budget; the per-handler cap (AC-R13f) is a per-individual fairness rule. In practice, most subscribers sit well below 1.0 ms; the per-handler cap catches single-subscriber regressions the chain-sum average would otherwise hide. Chain-sum failures (> 2.0 ms 99p) and per-handler failures (any one handler > 1.0 ms Release at 99p) are independently reported CI failures. Each knob's legal range is governed by its own row in the Pipeline Capacity Knobs table; there is no cross-knob invariant to preserve. The pre-2026-04-22-pass-2 binding-invariant framing is explicitly repealed. **Dev/Release CI profile selection (pass-3 BLOCKING #3 resolution + pass-4 BLOCKING #4 injection-mechanism spec)**: the `MONEYMINER_RELEASE_GATE` scripting-define-symbol governs which AC tier is active on a given CI build — BLOCKING CI Release gates (AC-R13c Release sub-budget, AC-R13f Release per-handler, AC-D1-release, AC-P2) compile-in only under that define via `#if MONEYMINER_RELEASE_GATE` lexical guards (NOT `[Conditional("MONEYMINER_RELEASE_GATE")]` — the attribute strips call sites not method bodies and interacts poorly with IL2CPP High stripping; `#if` is the authoritative pattern); Dev PLAYMODE tiers (AC-R13d, AC-R13f Dev, AC-D1-dev) compile-in under `#if UNITY_EDITOR || DEVELOPMENT_BUILD`. The CI pipeline sets the define per build variant (Release-artifact build vs. Development-build) so a Dev build cannot silently satisfy a Release BLOCKING gate.

**CI injection mechanism (pass-4 BLOCKING #4 resolution — named)**: Unity 6.3 LTS does NOT accept `-define` as a command-line argument to the test runner or builder. The define must be injected via Unity editor scripting invoked from CI. The project ships a `tools/ci/SetReleaseGate.cs` editor script (to be authored during Technical Setup sprint) exposing a static method `MoneyMiner.CI.ReleaseGateSetter.EnableReleaseGate()` that calls `PlayerSettings.SetScriptingDefineSymbols(NamedBuildTarget.Android, new[] { "MONEYMINER_RELEASE_GATE", …existing… })` (and the iOS equivalent). The `game-ci/unity-test-runner@v4` CI workflow invokes this via its `customParameters: -executeMethod MoneyMiner.CI.ReleaseGateSetter.EnableReleaseGate` pre-build step on Release-artifact jobs only; Dev jobs do NOT invoke it so the define stays absent. A complementary `DisableReleaseGate()` method is invoked as a post-job teardown to prevent define-set persistence bleeding across runs. Without this editor-script mechanism in place, the nine Release BLOCKING CI gates listed above cannot be verified at CI time — they would silently compile out and pass by absence. The Technical Setup sprint MUST land `SetReleaseGate.cs` before Input's first CI merge, or the Release BLOCKING gates are effectively ADVISORY. Flag for `/test-setup` skill when invoked.
- **Coupled — `occlusion_offset_dp` and `edge_padding_dp`**: on 19.5:9 portrait, the R7 clamp `min(touch.y + offset, safeArea.top − padding)` pins to the ceiling for touches within ~80 dp of the top. Raising offset without also reviewing padding can produce a "tool always spawns at the same vertical position regardless of where I tapped" effect on small devices. Review together on the minimum-spec target.
- **Independent — CI severities**: the R1 / R12 / R4 grep gates do not interact with each other or with the gesture classifier; they are architectural discipline, not perceptual tuning.
- **Independent — `PhaseTransitionSuppressDuration_ms` from all other knobs**: the phase-transition suppression timer is purely defensive and does not feed into any other formula. Tune based on playtest observation of ghost-tap incidents near phase boundaries on the reference device.

**Locked constants (not tuning knobs)**:
- `t1_latency_max` = 16.67 ms (registry, owned by Event Bus GDD; derived from 60 fps budget; any change is an architecture change crossing multiple GDDs)
- `t2_latency_max` = 100 ms (registry, owned by Event Bus GDD; Pillar 5 Feedback Floor; same constraint)
- Four-class gesture enum `{InProgress, Tap, Drag, LongPress}` (D.3; adding a fifth class is an ADR action, not a tuning change)
- First-touch-only (R2; repealing is an ADR action)
- Reference device authority = **ADR-003** (shared with Save/Load; Input inherits the same reference device selection rather than naming one inline)

## Visual/Audio Requirements

**Not applicable.** Input System has no runtime visual or audio presence — R8 explicitly forbids audio, visual, animation, haptic, and particle-spawn calls from `InputDispatcher.cs`. **Asset spec handoff**: there are no Input-owned assets. ### H.4 Performance & Platform

> **Shared-dataset preamble (pass-3 BLOCKING #6 resolution + pass-4 BLOCKING #6 tier-split clarification)**: AC-R13a (PreFire 99p), AC-R13b (T1Chain 99p), AC-R13c (99p of per-frame-sum), AC-R13d (Dev 99p), and AC-R13f (per-handler 99p) all operate on a **single shared paired dataset PER BUILD TIER** — 1000 synthetic touch events across 60 simulated frames, with every frame contributing a `(PreFire_ms, T1Chain_ms, {per-handler_ms × 4})` paired tuple. CI MUST compute the tier's ACs' 99ps from that tier's single recording; a separate capture run per AC within the same tier is not permitted.
>
> **Tier split (pass-4 BLOCKING #6 resolution)**: because AC-R13f's Release path uses inline `ProfilerMarker.Begin/End` (compiled under `#if MONEYMINER_RELEASE_GATE`) and its Dev path uses inline `Stopwatch.GetTimestamp` (compiled under `#if UNITY_EDITOR || DEVELOPMENT_BUILD`), a single physical CI build produces only one tier's instrumentation path. Therefore: **Release-tier ACs (AC-R13c, AC-R13f Release, AC-D1-release, AC-P2) share a single Release-artifact build's capture**; **Dev-tier ACs (AC-R13a, AC-R13b, AC-R13d, AC-R13f Dev, AC-D1-dev) share a single Dev-build's capture**. Cross-tier comparison (e.g., "does the Release 99p correlate with the Dev 99p?") is informational only, not a gate — the two datasets come from different build variants with different measurement overheads and are not directly comparable. Each tier's internal paired-per-frame correlation IS preserved within its own recording.
>
> **Per-event vs per-frame sampling (pass-4 RECOMMENDED-1 clarification)**: "1000 events across 60 frames" ≈ 17 events/frame. AC-R13a / AC-R13b / AC-R13f operate on the full ~1000-event series for their per-event 99p computation. AC-R13c operates on the 60-frame series for its per-frame-sum 99p computation (frame-aggregated: each frame contributes the SUM of that frame's event-level `(PreFire_ms, T1Chain_ms)` values). Both are drawn from the same tier-captured recording pass.
>
> Rationale: the three independent 99ps and the 99p-of-sum are only meaningfully related when drawn from the same frame samples within a tier — running them from independent captures within a tier would disconnect the per-frame correlation that AC-R13c's 99p-of-sum gate depends on, and would allow a frame with `(PreFire=2.1, T1Chain=1.9)` to be hidden in one run's capture while `(PreFire=1.8, T1Chain=2.1)` hides in another. When AC-R13a or AC-R13b fails but AC-R13c passes, the diagnosis is "sub-budget imbalance on this paired frame" (not a Pillar 5 breach); when AC-R13c fails but AC-R13a/b pass, the diagnosis is "worst-paired-frame jointly exceeds the Input main-thread sub-budget" — see AC-R13c's correlated-tail clause for the routing path. The CI reporter MUST label each 99p with the paired-frame index at which it was observed so triage can identify correlated tails.

- **AC-P1 — Feedback Floor End-to-End (Pillar 5, device-frame-rate-aware)** — GIVEN `input.tap.main` fires on the ADR-003 reference Android device, WHEN measured end-to-end as a **separate wall-clock** from touch `began` received by Unity through `Fire` → last T1 subscriber return → VFX/Juice render → render frame submission, THEN total wall-clock satisfies:

  - **60 fps target contract (Release shipping target):** total wall-clock ≤ **100 ms** when the device is sustaining 60 fps (frame budget 16.67 ms). 100 ms = `t2_latency_max` from `design/registry/entities.yaml` — Pillar 5 Feedback Floor at 60 fps target. Input contributes ≤ 4 ms (R13.c); ≥ 96 ms remains for VFX/Juice, HUD, other T1 channels, and render-frame-submission.
  - **30 fps floor contract (Mali-G52 thermal-throttled / low-end Android):** total wall-clock ≤ **133 ms** (100 ms + one 33.33 ms frame of render-submission slack) when the device is running at the 30 fps floor (frame budget 33.33 ms). Input's 4 ms sub-budget is unchanged at 30 fps (same main-thread cost); the 33 ms relaxation reflects render-submission latency at the floor. Below 30 fps is a thermal emergency outside this AC (see EC Classifier-1 + OQ-1).

  The frame-rate class of the test run is determined by `Application.targetFrameRate` and `UnityEngine.Time.unscaledDeltaTime` sampling in the harness (a rolling mean over a **wall-clock window of ≥ 500 ms** immediately preceding the measured tap — pass-4 BLOCKING #10 clarification: wall-clock framing replaces the prior 30-frame fixed-count window because 30 frames = 0.25 s at 120 Hz which is too narrow for thermal-stability sampling; 500 ms gives ≥ 15 samples at 30 fps, ≥ 30 at 60 fps, ≥ 60 at 120 Hz). `unscaledDeltaTime` is used in place of `deltaTime` so the rolling mean survives `Time.timeScale == 0` during paused scenes without pollution by `deltaTime = 0` samples. Runs observed at intermediate frame rates (e.g. 45 fps) assert against linear interpolation between the two contracts: `AC-P1_threshold_ms = 100 + (33 × (60 − fps) / 30)` for `30 ≤ fps ≤ 60`.

  - **`fps < 30` (thermal emergency, out of AC-P1 contract) with hysteresis (pass-4 BLOCKING #10)** — the harness MUST short-circuit the sample to `UNMEASURABLE`: the tap is excluded from the AC-P1 distribution and logged to a separate thermal-emergency counter. To prevent bimodal distribution pollution at the 30 fps boundary (a device oscillating 29/31 fps alternating UNMEASURABLE frames and 133 ms threshold frames), **hysteresis band applies**: once the rolling-mean fps drops below 30, the harness remains in UNMEASURABLE mode until the rolling mean reaches ≥ 32 fps for ≥ 10 consecutive 500 ms windows (≥ 5 s sustained recovery) before re-entering the in-domain measurement contract. This eliminates 29/31 fps oscillation bimodality. If `fps < 30` samples exceed 5 % of the run, the run is marked INCONCLUSIVE (not FAIL).
  - **INCONCLUSIVE CI reporter behavior (pass-4 BLOCKING #9 resolution)** — `Assert.Inconclusive(message)` in NUnit produces a `TestStatus.Inconclusive` result. The `game-ci/unity-test-runner@v4` XML reporter MUST be configured to treat INCONCLUSIVE as a **warning-level annotation (non-merge-blocking)** for AC-P1 specifically. The CI job emits a GitHub Actions warning annotation with payload `{ac: AC-P1, reason: thermal-emergency-sample-rate, device: {ADR-003-identifier}, run: {run-id}}` and routes the run to the performance-analyst Slack/email channel via the `notify-perf-analyst-on-inconclusive.yml` workflow (to be authored during Technical Setup sprint). The run does NOT fail CI; the annotation is the signal for OQ-1's post-MVP telemetry pass and OQ-8's thermal-inflation calibration. Without this configuration, INCONCLUSIVE runs silently pass (default NUnit-to-XML mapping) — that default is rejected here. Failing a thermal-emergency run would punish device hardware choices the AC doesn't cover; silently passing would hide a real device-tier performance regression. Warning-level annotation routed to a human is the adjudicated balance.
  - **`fps > 60` (120 Hz high-refresh devices)** — the applied threshold is clamped to 100 ms (the 60 fps target floor); the linear interpolation formula is NOT extrapolated above 60 fps. Rationale: the 60 fps contract already guarantees perceptual "instant" per Pillar 5; a 120 Hz device has more headroom, not a tighter obligation. A harness measuring at 120 fps asserts against 100 ms and records the device as "high-refresh, over-target" for downstream tuning insight.

If measured inflation is < 15 % (ceiling effectively > 4.6 ms), OQ-8 may propose tightening in a future revision; if measured inflation is > 25 % (the 5.0 ms ceiling is structurally insufficient), that is a hardware/optimization problem to address in Input implementation, not by relaxing the AC. **First-on-hardware calibration (OQ-8) is now an informational data-capture step, not a threshold-deriving step**: OQ-8 records measured inflation at first-run and logs it for post-MVP review; it does not gate AC-P3's ceiling. Mean-based and short-window (≤ 30 s) measurements are explicitly insufficient and MUST NOT be used to satisfy this AC — thermal-induced tail latency is the failure mode being gated here, and short windows / mean stats mask it. **fps-floor precondition**: if the rolling-mean fps (wall-clock ≥ 500 ms window, hysteresis per AC-P1) drops below 30 for any ≥ 5-consecutive-frame window inside the 120 s soak, the run is marked THERMAL-EMERGENCY (not PASS / not FAIL — surfaced to technical-director per AC-P1 out-of-domain rule). **CI operability note**: AC-P3's ≥ 120 s window exceeds typical per-commit PlayMode budgets; this AC runs in a dedicated soak-test job (nightly or pre-release, not per-commit) on the reference hardware. **DEVICE** (ADR-003)

- **AC-P4 — Aspect-Ratio Safe-Area Coverage** — GIVEN simulated safe-area rects for 19.5:9 portrait (iPhone with dynamic island) and 21:9 (tall-narrow Android), WHEN `DeviceDisplayMetrics` is queried for each configuration, THEN `safeArea.rect` values differ between configurations (platform-correct insets, not a constant stub); the occlusion clamp `min(touch.y + occlusion_offset_dp, safeArea.top − edge_padding_dp)` produces different ceiling values per configuration. **PLAYMODE** (simulated) + **DEVICE** (ADR-003 for hardware extremes)

### H.5 Consumer Integration

- **AC-INT1 — Tool System Polling Contract** — GIVEN Tool System polls `InputState.CurrentPointer` in `Update()` during an active drag (R3.d — no `input.drag.update` event exists), WHEN the pointer position changes between frames, THEN each successive `Update()` reads a fresh `ScreenPosition` reflecting the latest touch position; no Event Bus subscription is required for per-frame drag preview. **PLAYMODE**

- **AC-INT2 — Collection System Tap-with-Target** — GIVEN a world collider on a tap-eligible layer is at the touch coordinate, WHEN `input.tap.main` fires, THEN `TapPayload.Target` is the collider's `GameObject`; Collection System's subscriber receives a non-null `Target` without performing a second Physics2D query of its own (R4 two-pass resolution is Input's responsibility). **PLAYMODE**

- **AC-INT3 — Modal Suppression Lifecycle** — GIVEN `PushModalSuppression(modalId)` is called (modal opens), THEN touch events are suppressed immediately; GIVEN `PopModalSuppression(modalId)` at depth = 0, THEN touch events resume in the next Update frame (no events fire in the frame `Pop` is called); nested modals (`Push` × 2 then `Pop` × 1) keep depth ≥ 1 and suppression continues. **PLAYMODE**

- **AC-INT4 — Save/Load Suppression Lifecycle** — GIVEN `SetLoadSuppression(true)` is called at `LoadingFromDisk` entry (Save/Load calling into Input directly), THEN all touch input is suppressed; GIVEN `SetLoadSuppression(false)` is called at `LoadingFromDisk` exit, THEN input resumes in the next Update (provided modal depth = 0 and no phase-transition timer is active per R10.d priority). **PLAYMODE**

- **AC-INT5 — Phase-Transition Suppression Window** — GIVEN `gameplay.level.phase_changed` fires on Event Bus (e.g., Prep → Rush), WHEN Input enters `Suppressed.PhaseTransition`, THEN no touch events fire for `PhaseTransitionSuppressDuration_ms` ms; after the timer elapses (and modal depth = 0, load suppression = false), Input returns to `Active` and the next touch fires normally. **PLAYMODE**

### H.6 Invariants (What Must NOT Happen)

- **AC-INV1 — No Tap Synthesized from Drag** — GIVEN any gesture that at any point satisfied the Drag classification condition (`d ≥ d_slop` OR early-drag gate `d ≥ d_early AND t ≥ t_early`), WHEN `ended` fires, THEN `input.tap.main` is never published for that gesture across all `(d, t, v)` combinations; Event Bus tap-publish count for the gesture lifetime = 0. **PLAYMODE**

- **AC-INV2 — No Events in Any Suppressed State** — GIVEN `State` is any `Suppressed.*`, `Uninitialized`, or `Disposed`, WHEN any number of touch events are delivered, THEN Event Bus publish count for `input.tap.main`, `input.drag.start`, `input.drag.end`, `input.gesture.longpress.unhandled`, `input.contact.began` (pass-4 BLOCKING #13 addition), AND `audio.input.longpress.received` (pass-4 BLOCKING #11 addition) = 0; `CurrentPointer.IsPressed = false`; `CurrentPointer.ScreenPosition` holds last pre-suppression value (never `Vector2.zero`). **PLAYMODE**

- **AC-INV3 — No drag.end Across Pause Boundary** — GIVEN a drag in-flight at `OnApplicationPause(true)`, WHEN pause fires and then `OnApplicationPause(false)` fires, THEN `input.drag.end` does NOT fire at any point; Unity's synthetic post-resume `ended` for the stale touch produces no event; zero ghost drag.end events cross the pause boundary. **PLAYMODE**

- **AC-INV4 — No Event Bus Subscription Leak** — GIVEN Input subscribes via `SubscribeAutoManaged<PhaseChangedPayload>` (Event Bus auto-managed lifecycle), WHEN `InputDispatcher` is destroyed (scene unload) and the scene reloads, THEN all `input.*` and Gameplay namespace subscriptions from the prior instance are unregistered before the next scene's `Awake` runs; Event Bus debug subscriber count shows zero leaked handles. **PLAYMODE**

- **AC-INV5 — Input Structural Cleanliness (Summary Gate)** — GIVEN CI runs on any commit touching `src/Input/`, WHEN AC-R1 (InputSystem allowlist grep) AND AC-R12 (gameplay namespace grep) AND AC-R4-nonalloc-grep (allocating OverlapPoint banned) all pass, THEN the Input module is confirmed structurally clean: zero non-allowlist Unity Input System access, zero gameplay coupling, zero allocating Physics2D overloads. This invariant is a sprint-review summary signal — it passes iff all three component ACs pass. **BLOCKING CI**
