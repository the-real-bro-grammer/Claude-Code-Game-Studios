## Summary

Input System is Money Miner's touch dispatcher ŌĆö the single authority that converts raw Unity touches into typed, tier-budgeted events for every consuming system. It owns the tap-vs-drag classifier, the safe-area-aware pointer state, the finger-occlusion offset that keeps tools visible above the thumb, and the input-suppression gate that prevents ghost taps during PrepŌåÆRush phase transitions or modal dialogs. Every system that reads player intent (Tool System's drag-to-place, Collection System's tap-to-catch, Biome Map's swipe-to-pan, HUD's button taps) consumes Input's events ŌĆö no system reads `UnityEngine.InputSystem` APIs directly.

> **Quick reference** ŌĆö Layer: `Foundation` ┬Ę Priority: `MVP` ┬Ę Key deps: none (zero-dep foundation) ┬Ę Publishes on Event Bus: `input.tap.main`, `input.drag.start`, `input.drag.end` (T1 Feedback), `input.gesture.longpress.unhandled` (T2 Feedback, HUD REQUIRED subscriber per R3.c + AC-R3d-required + AC-R3d-audio ŌĆö dual-channel at MVP: offset visual tick above thumb + audio tick via `audio.input.longpress.received`; Haptics REQUIRED at VS as third-channel cert-gating addition per OQ-12 deaf-blind gap), `input.contact.began` (T2 Feedback, **HUD REQUIRED subscriber on every Rush-phase-capable scene** per R3.e pass-5 + AC-R3g-required; opt-in on all other scenes)

## Overview

Mobile touch input looks trivial until you ship it. A "tap" is not a single Unity event ŌĆö it is a `began` ŌåÆ `moved` ŌåÆ `ended` sequence that must be classified as tap vs drag vs hold within a bounded time/distance window. A "drag" that starts on a tool icon must not double-fire as a "tap" on the playfield beneath. A finger hovering over the bottom third of the phone must not occlude the critical tool it is placing. A tap during a Modal/Dialog overlay must be suppressed. A player whose phone is a Samsung Galaxy S-series at 20:9 aspect ratio must see the same hit-zones as a player on a 19.5:9 iPhone. These are the problems Input System owns ŌĆö and every consuming system downstream trusts Input to have already solved them before its handler runs.

The system provides four capabilities: (1) **typed event publication** ŌĆö Input subscribes to `UnityEngine.InputSystem.EnhancedTouch.Touch.activeTouches` on the main thread and fires typed events (`TapPayload`, `DragPayload`) on Event Bus Feedback channels at T1 tier, plus `LongPressUnhandledPayload` on a T2 channel (opt-in); Input claims a Ōēż 4 ms main-thread sub-budget inside the 16.67 ms frame (R13.c / D.1) so consumers can gate same-frame feedback while leaving Ōēź 12.67 ms for render, physics, and other systems; (2) **pointer state query** ŌĆö a pure `InputState.CurrentPointer` struct (position, pressed, age) that any gameplay system can read in `Update` / `FixedUpdate` without subscribing to events, for systems that need continuous state (Tool System's drag-to-place preview) rather than discrete events; (3) **phase-aware suppression** ŌĆö Input queries the current Level phase (Prep / Rush / Paused) and Modal/Dialog stack depth, suppressing or routing events accordingly so ghost taps cannot fire during phase transitions, pause, or modal overlays; (4) **platform safe-area and hit-inflation contracts** ŌĆö every published screen coordinate is clipped to the device safe area (notch + home indicator + punch-hole camera islands), and the 44pt iOS / 48dp Android minimum hit-target rule plus the 1.4├Ś visual-hitbox inflation rule (per `.claude/docs/technical-preferences.md`) are enforced at the Input layer so every consumer inherits correct hitboxes without duplicate logic.

The system enforces five rules: **no direct `UnityEngine.InputSystem` access outside Input** (CI-enforced grep ŌĆö consumers read only `InputEvent.*` events and `InputState.CurrentPointer`); **no input during `LoadingFromDisk`** (Save/Load holds the pause gate; Input's suppression mirrors the `SaveManager` state); **no tap synthesis from drag** (a drag that ends within the tap-classification window still fires `input.drag.end`, never `input.tap.main` ŌĆö downstream systems that want either must subscribe to both); **no main-thread blocking** (Input's `Fire` ŌåÆ handler chain completes synchronously within Input's 4 ms main-thread sub-budget per R13.c; any consumer whose handler would overrun the 0.5 ms per-handler threshold must defer via `Task.Run` or the next frame via `EventBusPlayerLoopSystem`); **no gameplay rules in Input** (Input publishes raw pointer intent; whether a tap *hits* a coin is Collection System's concern, whether a drag *places* a ramp is Tool System's concern, whether a tap *advances* a dialog is Modal/Dialog's concern).

This GDD defines the design contract ŌĆö events, states, suppression rules, tuning thresholds. Unity Input System package configuration (Input Actions asset structure vs `EnhancedTouch` direct polling; dispatch mode choice between `PlayerInput` component, generated C# class, or `InputAction` callbacks) is an implementation detail scoped for a future ADR when MVP implementation begins. Consumers designing against Input System may rely on the contract described here.

## Player Fantasy

Input System has no player-facing fantasy ŌĆö it is the channel through which every other system's fantasy arrives. No player will ever say "I love this game's touch handler." They will say "that ramp snapped exactly where I wanted it" or "the gopher went for the bait the instant I tapped," and this is the system that made those sentences true.

The fantasy Input serves is Pillar 5's promise ŌĆö **No Input Goes Silent**. Every tap and drag must reach its consuming system (Tool, Collection, HUD, Biome Map) within a single frame so that audiovisual feedback can fire inside the 100ms window the player's nervous system treats as "instant." Input is the first 16.67ms of that budget. Everything downstream ŌĆö the tool spawn, the coin pop, the squeak ŌĆö depends on this system getting out of the way fast enough that the other systems have time to be satisfying.

Picture the moment a player cares about most: Rush phase, 18 seconds on the clock, a gopher stream bending toward the left ledge, and the player drags a ramp from the tray with their right thumb. Their finger is directly over the ramp. They lift. The ramp is exactly where they meant it ŌĆö offset cleanly above the point of contact, snapped to the grid, with a satisfying *clack* inside 100ms. They never thought about their finger. They thought about the gophers.

That is the fantasy Input protects ŌĆö and it is not Input's fantasy. It belongs to Tool System, to Collection, to the PrepŌåÆRush handoff. Input's job is to be the channel that delivers the tap and drag so faithfully that the player never notices Input exists. The tap-vs-drag classifier, the 1.4├Ś hitbox inflation, the spawn-above-finger offset, the suppression gate that kills stale taps crossing the PrepŌåÆRush boundary ŌĆö every one of these exists to keep the player's attention on the board and off their hand.

The fantasy is NOT "my tap was registered." The fantasy is "I never had to wonder whether my tap was registered." Silence when working, invisible when working. When Input is doing its job, the player is thinking about gophers, not fingers.

## Detailed Design

### Core Rules

**R1. Single input authority.** No code outside `InputDispatcher.cs` and `InputState.cs` accesses `UnityEngine.InputSystem.*` namespaces (including `EnhancedTouch`, `LowLevel`). Consumers read only `InputEvent.*` typed events on Event Bus and the `InputState.CurrentPointer` struct. Enforcement ŌĆö CI grep pattern `^using UnityEngine\.InputSystem(\.EnhancedTouch|\.LowLevel)?;` applied to all `.cs` files with allowlist: `src/Input/InputDispatcher.cs`, `src/Input/InputState.cs`, `tests/unit/Input/**/*.cs`, `tests/integration/Input/**/*.cs`. Any match outside the allowlist fails CI.

**R2. First-touch only at MVP.** Input tracks the single active touch identified by the finger ID of the first touch to enter `began` phase. Any simultaneous second-or-higher-finger contact is silently ignored (no event published, no log). The tracked finger is released on `ended` or `canceled`. Multi-touch (pinch-to-zoom on Biome Map, dual-thumb gestures) is explicitly post-MVP and requires an ADR to repeal R2. Enforcement ŌĆö PlayMode test: simulated two-finger simultaneous `began` produces exactly one event sequence (from the first finger only).

**R3. Gesture classification.**

- **R3.a Tap.** Movement delta `< 10pt` (iOS) / `< 10dp` (Android) for the full touch duration AND total duration `Ōēż 300ms`. Published as `input.tap.main` on the `ended` phase ŌĆö **never on `began`**. A tap-that-becomes-a-drag generates zero taps. Late-lift forgiveness: a touch lifting at 301ms+ with `< 10pt` movement and no velocity is still classified as a tap (motor-impaired accommodation).
- **R3.b Drag.** Movement delta `Ōēź 10pt` / `10dp` at any time during the touch, OR delta `> 6pt` / `6dp` after 150ms (slow-deliberate-drag early trigger, critical for Rush-phase ramp placement where waiting 300ms for classification wastes 18% of the 100ms feedback window). On the frame the threshold is crossed, `input.drag.start` fires with the original `began` position as `StartScreen` and the current position as `EndScreen`. On `ended`, `input.drag.end` fires with the original `began` position and the final position.
- **R3.c Long-press ŌĆö reserved at MVP, published as unhandled with REQUIRED MVP dual-channel subscriber.** Duration `> t_tap_max` with `d < d_slop` and lift-velocity `v Ōēź v_lateLift` is classified as LongPress. Input publishes `input.gesture.longpress.unhandled` on a T2 Feedback channel (not T1) with payload `LongPressUnhandledPayload(ScreenPosition, DurationMs, Target?)`. **HUD is a REQUIRED MVP subscriber** ŌĆö HUD registers on scene start and, on every LongPress classification, renders a **dual-channel feedback contract** (pass-3 BLOCKING #1 resolution ŌĆö `ScreenPosition` under-the-thumb occlusion failure):
    1. **Visual tick (occlusion-offset)** ŌĆö sprite rendered at `ScreenPosition + InputState.GetOcclusionOffset(ScreenPosition)` (i.e. `+72 dp` upward clamped to `safeArea.top ŌłÆ 8 dp` per R7), so the feedback appears above the thumb-pad that just long-pressed (not under the finger still in contact). Minimum visual spec handed to HUD GDD: sprite diameter Ōēź 48 dp (matches the accessibility hit-target floor), duration Ōēł 200 ms dim/fade, **minimum 3:1 contrast against the expected HUD/biome background** (art-director to define the canonical sprite in HUD GDD / asset spec; default shipping treatment is a flat neutral-grey disc with thick outline consistent with the Cartoon Heist Diorama art direction).
    2. **Audio tick (Pillar 5 non-visual publish-path fail-safe at MVP)** ŌĆö a short audio cue published on Event Bus Feedback channel `audio.input.longpress.received` (4-segment slug conforming to Event Bus R5 regex `^(audio|vfx)\.[a-z_]+\.[a-z_]+\.[a-z_]+$` ŌĆö namespace corrected in pass-4 BLOCKING #11 from the pass-3 draft `input.longpress.received`, which would have namespace-collided with Input's own `input.*` Feedback channels). HUD subscribes to this channel and routes it through Audio Bus for playback; the canonical audio asset tag is owned by the Audio Bus GDD (pending authoring). The audio tick is timed to the same 200 ms window as the visual tick. **Language scope (pass-5 BLOCKING #2 resolution):** the audio channel is the **publish-path** fail-safe ŌĆö Input publishes, HUD subscribes, HUD routes through Audio Bus. AC-R3d-audio asserts the publish side of that chain (BLOCKING CI); the downstream InputŌåÆHUDŌåÆAudio BusŌåÆdevice-audio end-to-end chain is not asserted at MVP because Audio Bus GDD is unauthored and a DEVICE-tier end-to-end AC would block on undesigned downstream infrastructure. The prior pass-3/4 "guaranteed non-visual fail-safe" framing is **explicitly downgraded** to "publish-path fail-safe" per pass-5 game-designer BLOCKING #2 ŌĆö the guarantee applies to what Input controls (the publish contract), not to what Input does not control (downstream rendering / routing / device audio). **Deaf-blind combined-failure gap (pass-5 BLOCKING #18 / OQ-12):** the dual-channel label assumes the two channels fail independently; for a deaf-blind user both channels fail simultaneously. Haptics is the third natural mobile channel; at MVP, Haptics remains a VS-tier subscriber (see Section F Downstream phased table) and the deaf-blind combined-failure gap is a known regression tracked in OQ-12. At VS, Haptics subscription to `input.gesture.longpress.unhandled` is REQUIRED and cert-gating for the third channel.
  This dual-channel-at-MVP contract is the minimum needed to honour Pillar 5's "no input goes silent" promise for users who can perceive at least one of visual or audio; a single visual-only tick at raw `ScreenPosition` (the pass-2 spec) fails the perceptual contract for motor-impaired players still in contact at lift, and is superseded by this spec (see review log 2026-04-22 pass-3 BLOCKING #1). At VS, Accessibility Service may extend or replace either channel with a richer treatment; Haptics joins as the REQUIRED third channel; the combined deaf-blind gap closes. Post-MVP systems that require long-press semantics can subscribe alongside HUD without repealing R3.c. `input.tap.main`, `input.drag.start`, and `input.drag.end` never fire for a LongPress-classified gesture. Enforcement ŌĆö CI asserts Ōēź1 T2 subscriber is registered at scene start on any scene that spawns `InputDispatcher` (see AC-R3d-required). The HUD GDD is the owner of the dual-channel implementation spec; this GDD defines the contract, not the sprite asset or audio-tag filename.
- **R3.d Continuous drag state via polling, not events.** No `input.drag.update` event exists. Systems needing per-frame drag position (Tool System's drag-to-place preview) read `InputState.CurrentPointer` in `Update`. Rationale: per-frame bus traffic at 60fps would blow T1 subscriber-handler budgets; polling is cheaper and self-throttles to each consumer's own frame cadence.

- **R3.e Contact-Began event ŌĆö REQUIRED subscriber on Rush-phase-capable scenes (pass-5 BLOCKING #1 resolution ŌĆö promotes pass-4 advisory framing).** On the `began` phase of every tracked touch (subject to R10 suppression), Input publishes `input.contact.began` on a T2 Feedback channel with payload `ContactBeganPayload(Vector2 ScreenPosition, int FingerID, float BeganRealtimeSeconds)`. **Subscription tier (pass-5 game-designer + ux-designer convergence, CD-adjudicated):**
    - **REQUIRED MVP subscriber on Rush-phase-capable scenes** ŌĆö every scene that spawns `LevelRuntime` (i.e., every gameplay scene with a Rush phase: level-play scenes, stress-scene prototypes) MUST register HUD as a T2 subscriber on `input.contact.began` rendering a minimal on-contact pending visual (a brief "contact received" dim marker at the occlusion-offset position per R7; Ōēź 24 dp diameter; Ōēż 100 ms fade; Ōēż 3:1 contrast against expected HUD/biome background per WCAG 2.1 SC 1.4.11 Non-text Contrast) that extinguishes on the subsequent `input.tap.main` / `input.drag.start` / `input.gesture.longpress.unhandled` for the same FingerID. AC-R3g-required (new BLOCKING CI per pass-5 B1) asserts this subscription via behavioral fixture mirroring AC-R3d-required.
    - **Opt-in on non-Rush-phase scenes** ŌĆö title, biome-map, settings, pause, modal-only scenes MAY subscribe HUD or VFX/Juice for on-contact feedback but are not required to. These scenes have no Rush-phase sub-600 ms silence window (the load-bearing Pillar 5 case is Rush-phase quick-response play); the perceptual cost of on-contact silence at the title screen is negligible against the cost of authoring pending-state visuals in every UI scene.
  Rationale: R3.a requires tap to fire on `ended` (preserves tap-vs-drag disambiguation per D.3) but that produces up to 300 ms / 600 ms of on-contact silence before any T1 event reaches subscribers ŌĆö a structural Pillar 5 regression for the slow-tap / accommodation subpopulation during Rush phase. Pass-4 deferred the closure as advisory-only; pass-5 game-designer + ux-designer convergence flagged advisory-only as structural paper resolution recurring the pass-2 T2-LongPress pattern. Pass-5 Phase 6A resolves via Rush-phase REQUIRED promotion (scope discipline: close the gap where Pillar 5 is load-bearing, not everywhere). The classifier's Tap / Drag / LongPress outputs are unchanged ŌĆö R3.e fires IN ADDITION to whatever the classifier eventually resolves. Subscribers MUST correlate `ContactBeganPayload.FingerID` with the subsequent event's FingerID to extinguish their pending-state UI (the FingerID is added to `TapPayload` / `DragPayload` / `LongPressUnhandledPayload` if not already present ŌĆö flag for consistency-check). If R10 suppression is active at `began`, `input.contact.began` is NOT published (mirrors AC-INV2 invariant ŌĆö no events in Suppressed state, extended to T2 `contact.began` channel). Per-frame budget impact: one additional T2 publish per gesture lifetime (at `began`) ŌĆö negligible against the T2 100 ms Pillar 5 envelope and not counted against Input's 4 ms main-thread sub-budget (T2 dispatch is asynchronous from R13.c's main-thread measurement ŌĆö confirm at Event Bus consistency check). **Event Bus impact**: one new row in Event Bus Interactions table (Emitter: Input System, Event: `input.contact.began`, Namespace: Feedback, Payload: `ContactBeganPayload`, Subscribers: HUD REQUIRED on Rush-phase scenes / HUD + VFX/Juice opt-in elsewhere, Latency: T2) ŌĆö pending Event Bus GDD amendment at Input pass-5 Phase 6C merge.

Enforcement ŌĆö unit tests for each branch (at-threshold, one unit below, one unit above); integration test confirming no `input.tap.main` fires after a drag; AC-R3g-required (new BLOCKING CI pass-5 B1) asserts HUD is registered as non-fixture subscriber on Rush-phase-capable scenes; AC-R3g (PLAYMODE, retained from pass-4) asserts `input.contact.began` publishes on every tracked `began` under `Active` state (suppression-aware).

**R4. Hit-target resolution order.** For `TapPayload.Target`, Input performs a two-pass resolution on the main thread:

1. **UI raycast** ŌĆö Clear the pre-allocated `List<RaycastResult>`, populate `PointerEventData` with the touch's safe-area-local screen position, and call `EventSystem.current.RaycastAll(pointerEventData, results)` (caller-allocated list, zero-alloc on the Input side). If `results.Count > 0` and `results[0].gameObject != null`, treat the first result as `Target` and SKIP Pass 2. **Presence of raycast results is the authoritative UI-hit signal.** Do NOT call `EventSystem.IsPointerOverGameObject(int touchId)`: when `InputSystemUIInputModule` is the active UI input module (standard under Unity 6.3's new Input System), the `touchId` from `EnhancedTouch.Touch.touchId` does not map to the EventSystem's internal pointer IDs ŌĆö the module synthesises its own pointer IDs and does not pass raw OS touch IDs through. Using the overload produces unreliable results. **UNVERIFIED ŌĆö engineering task during implementation**: confirm that `PanelRaycaster` (UI Toolkit) participates in `EventSystem.RaycastAll` results in Unity 6.3 LTS; if it does not, UI Toolkit panels would fall through to Pass 2 (world raycast) and the R4 "UI first" ordering would break for any UI Toolkit content. Add a PlayMode test with a UI Toolkit panel over a world collider on first implementation and promote this caveat to a resolution before merge.
2. **World raycast** ŌĆö `Physics2D.OverlapPointNonAlloc(worldPoint, resultsBuffer, layerMask)` with pre-allocated 8-element buffer and layer-filtered collision matrix. First hit wins. `Physics2D.OverlapPoint` (allocating) is BANNED ŌĆö CI grep enforces this.

If both passes return null, `Target = null` (playfield tap). Rationale ŌĆö reversing the order would let a tap pass through a modal overlay and reach the playfield beneath (ghost tap bug). ScreenSpaceOverlay canvases are outside the camera frustum and cannot be detected by Physics2D queries; EventSystem raycast is the only correct UI path. Enforcement ŌĆö PlayMode test: UI layer over a world collider; published `Target` is the UI object.

**R5. Coordinate contract ŌĆö safe-area-local pixels.**

- **R5.a Safe-area clipping.** Touches whose `began` position lies outside `Screen.safeArea` (iOS notch, home indicator, Android punch-hole exclusion zones, back-gesture edge) are suppressed at source ŌĆö no event published, no `CurrentPointer` update. Publishing-and-filtering at the consumer layer creates a race where fast taps reach T2 subscribers before their own filter runs.
- **R5.b Every published `Vector2` is safe-area-local.** Origin is `Screen.safeArea.bottomLeft`. Consumers do NOT add their own safe-area offset. The safe-area rect is cached once on scene load (it does not change during a session on any supported iOS/Android device). A touch that begins inside safe area and moves outside remains tracked; positions are clamped to the safe-area boundary on each `Update`.

Enforcement ŌĆö unit test with mock safe-area rect: touch position outside ŌåÆ no event; edge-crossing touch ŌåÆ coordinates clamped.

**R6. Hitbox inflation is consumer-authored, not Input-enforced.** Input publishes raw finger contact coordinates. Consuming systems (Tool System, Collection System, HUD, Biome Map, Modal/Dialog) are responsible for authoring their colliders and UI RectTransforms with the 1.4├Ś visual-sprite-bounds inflation and the 44pt iOS / 48dp Android minimum hit-target floor (per `.claude/docs/technical-preferences.md`). Rationale ŌĆö a centralized `InflatedBoundsRegistry` at the Input layer would require every dynamic spawn/despawn (gophers mid-Rush, consumables mid-deploy) to register/deregister with Input, impacting the 60fps frame budget and adding state coupling that Unity's native collider model does not require.

- **R6.a `DeviceDisplayMetrics` singleton.** Input exposes a read-only `DeviceDisplayMetrics` struct (pixels-per-point, pixels-per-dp, safe-area rect, screen aspect ratio) populated in `Initialize()` before any touch event fires. Consumers reference this to derive correct sizes without platform `#ifdef`s. Registered as a cross-system constant in `design/registry/entities.yaml` (post-authoring Section 5b).
- **R6.b Debug-build floor assertion.** In `UNITY_EDITOR` and `DEVELOPMENT_BUILD`, every consuming system's UI element / collider registered through the generic `IInteractable.RegisterHitBounds(Rect)` helper asserts that the bounds width AND height meet the accessibility floor. Violations produce a blocking assertion (not a silent clamp ŌĆö silent clamps hide authoring errors). Release builds omit the assertion.

**R7. Finger-occlusion offset for spawn-above-finger.** When a consuming system (Tool System) requests a spawn position for an object dragged under the thumb, Input exposes `InputState.GetOcclusionOffset(Vector2 touchPoint) ŌåÆ Vector2`. Default offset: `+72dp` on the Y axis (Ōēł1.5├Ś a 48dp average thumb-pad diameter). If `touchPoint.y + 72dp > Screen.safeArea.top`, clamp to `Screen.safeArea.top - 8dp` (breathing room from ceiling). Fixed-offset over screen-relative ŌĆö at 19.5:9 vs 21:9, screen-relative offsets feel inconsistent. Tuning range in Section G.

**R8. Input produces zero feedback.** No sounds, visuals, animations, haptics, or particle spawns originate from `InputDispatcher`. Input publishes a typed event on Event Bus and returns. All feedback is the subscribing systems' concern (HUD for button flash, VFX/Juice for tap pop, Audio Bus for tap SFX, Haptics for rumble). Enforcement ŌĆö design-review gate: `InputDispatcher.cs` imports zero audio/visual/haptics namespaces; no `Play*()`, `Instantiate()`, `UnityEngine.Rendering.*`, `Handheld\.Vibrate`, `AndroidJavaObject`, `AndroidJavaClass`, `CoreHaptics`, `VibrationEffect`, or `\.Haptics\b` calls. (The prior grep `UnityEngine.InputSystem.XInput.Haptics.*` was replaced per review log 2026-04-22 BLOCKING #5 ŌĆö that pattern is gamepad-only and never matches on touch-only mobile builds. The new pattern covers `Handheld.Vibrate` Unity built-in, iOS CoreHaptics, Android JNI vibration paths via `AndroidJavaObject` / `AndroidJavaClass`, and generic `.Haptics` namespace fragments. Over-inclusive by design; false positives are acceptable in `InputDispatcher.cs` since any JNI usage there is itself suspect.)

**R9. MVP accessibility floors ŌĆö configurable, UI deferred to VS.** Input reads an `AccessibilitySettings` struct from `PlayerPrefs` on `Initialize()` with four configurable values (fourth added per pass-6 Phase 6D accessibility-specialist RECOMMENDED promotion ŌĆö `hit_target_scale` moved from design assumption to user accommodation):

| Setting | MVP Range | Default | VS UI |

**R10. Suppression gate ŌĆö no events in any Suppressed state.** When `InputDispatcher.State` is `Uninitialized`, `Disposed`, or any `Suppressed.*`, all touches are discarded: no `input.*` events fire, `InputState.CurrentPointer.IsPressed == false`, `InputState.CurrentPointer.ScreenPosition` returns its last valid value (NOT zero ŌĆö avoids consumers snapping to origin). Any in-flight gesture (begun but not classified) at suppression entry is silently abandoned; its eventual `ended` fires no event.

- **R10.a Phase suppression.** Input subscribes to `gameplay.level.phase_changed` on Event Bus (Gameplay namespace, T2 tier). On any `PrepŌåÆRush`, `RushŌåÆPaused`, or `RushŌåÆComplete` transition, Input enters `Suppressed.PhaseTransition` for `PhaseTransitionSuppressDuration` ms (Section G knob). Input does NOT query Level Runtime directly ŌĆö Level Runtime is a downstream consumer of Input; subscribing to the bus preserves zero-upstream-dependency.
- **R10.b Modal suppression.** Modal/Dialog calls `InputDispatcher.PushModalSuppression(modalId)` and `.PopModalSuppression(modalId)` directly (not via Event Bus ŌĆö modal open/close must guarantee suppression before the next frame's touch processing). Depth-counted ŌĆö nested modals do not prematurely lift suppression.
- **R10.c Load suppression.** Save/Load calls `InputDispatcher.SetLoadSuppression(bool)` at `LoadingFromDisk` enter/exit. Mirrors Save/Load's state machine (see save-load.md Section C, States and Transitions).
- **R10.d Priority.** `Suppressed.Loading` > `Suppressed.Modal` > `Suppressed.PhaseTransition`. All three are independent flags; `Active` is entered only when all three are simultaneously inactive. Debug-log state name is the highest-priority active flag.

**R11. Pause gesture cancellation (show-stopper fix).** On `OnApplicationPause(true)`: Input immediately abandons any in-flight gesture state, sets `_activeTouchId = -1`, and transitions to whatever suppression state is appropriate (Loading if Save/Load signals it, else a temporary resume-guard state). No `input.drag.end` fires on pause even if a drag was active ŌĆö the pause itself is the cancel signal. On `OnApplicationPause(false)`: Input resets `_activeTouchId = -1` before the first post-resume `began` is processed. This prevents the Unity Input System's post-resume synthetic `ended` event (the OS reports finger-up state on resume; Unity fires `ended` against the pre-pause touch state) from firing a stale `drag.end` on a gesture the player already abandoned. Enforcement ŌĆö PlayMode integration test: begin drag ŌåÆ trigger `OnApplicationPause(true)` ŌåÆ `OnApplicationPause(false)` ŌåÆ verify no `drag.end` fires; verify new touch after resume fires cleanly.

**R12. No gameplay logic in Input.** `InputDispatcher.cs` and `InputState.cs` do not `using` any of `MoneyMiner.Gameplay`, `MoneyMiner.Tools`, `MoneyMiner.Collection`, `MoneyMiner.Economy`, `MoneyMiner.Progression`. Input does not inspect hit-target components to decide event payload content (e.g., Input does NOT check "is this a coin?" to route to a different event). Whether a tap hits a coin, a ramp, a gopher, or a menu button is entirely the subscriber's concern. Enforcement ŌĆö CI grep on `using` directives.

**R13. Input Pipeline Budget.** Input's dispatch runs on the main thread inside a single 16.67 ms frame and MUST consume only a fraction of that budget so the frame can also render, step physics, and run gameplay. Input's main-thread sub-budget is layered:

- **R13.a Pre-Fire pipeline Ōēż 2 ms (Release) / Ōēż 2.5 ms (Dev).** From `EnhancedTouch.Touch.activeTouches` read through tap/drag classifier through `EventSystem.RaycastAll` + `Physics2D.OverlapPointNonAlloc` through `TapPayload` / `DragPayload` construction to the `EventBus.Fire(channel, payload)` entry. Measured inside `InputDispatcher.Update()`.
- **R13.b T1 subscriber chain Ōēż 2 ms (Release) / Ōēż 2.5 ms (Dev).** From `Fire` entry through last subscriber return on any `input.*` channel. This is Input's *slice* of Event Bus's T1 dispatch envelope ŌĆö the registry `t1_latency_max = 16.67 ms` (event-bus.md D.1) is the Event Bus's *per-channel* T1 window; Input claims 2 ms of it so the remaining Ōēź14 ms is available to other T1 channels firing in the same frame. **Cross-GDD aggregate coordination (pass-2 BLOCKING #4 resolution):** Input's 4 ms main-thread sub-budget (R13.c) is further constrained by Event Bus's `T1_per_frame_aggregate_ms = 8 ms` locked constant (event-bus.md D.1, pass-4 cross-coordination amendment 2026-04-22) ŌĆö a *per-frame sum across all T1 channels combined*. Input's 4 ms claim is 50% of that 8 ms aggregate; remaining Ōēż 4 ms is shared among all other MVP T1 producers (Collection, Tool, Hazard, Critter AI). **Rush-phase peak contention (pass-5 BLOCKING #6 resolution + OQ-13):** the 4 ms remainder is SHARED-contention across 4 Rush-phase T1 producers (Collection + Tool + Hazard + Critter AI); nominal per-producer share is 1 ms under uniform distribution, but Rush-phase peak frames (gopher-cluster burst + simultaneous tool placement + active hazard + critter-AI replanning) can drive per-producer demand above 1 ms for one or more producers on the same frame. If aggregate T1 demand for a peak frame exceeds 8 ms, Event Bus AC-F4-aggregate fails; the failure attribution is per-channel (not Input's failure unless Input's own 4 ms is exceeded). OQ-13 tracks empirical Rush-peak aggregate validation against the "Stress Scene" prototype (5 holes ├Ś 40 Rigidbody2D per technical-preferences.md pre-production action item #5); if measured Rush-peak aggregate exceeds 8 ms, the options are (a) retune Input's claim downward (breaks Input's 4 ms sub-budget), (b) retune the aggregate upward (breaks Event Bus's 16.67 ms per-channel window), or (c) move one or more Rush-phase T1 producers to T2 or poll-based consumption. OQ-13 flags the empirical-measurement trigger for that retune decision. The aggregate is enforced by Event Bus AC-F4-aggregate; any cross-producer breach is an Event Bus CI failure that attributes cost per channel, not an Input-only failure.
- **R13.c Input main-thread total Ōēż 4 ms (Release) / Ōēż 5 ms (Dev).** R13.a + R13.b. Leaves Ōēź12.67 ms of the 60 fps frame for render, physics, gameplay updates, and other T1 channels firing. **This is a sub-budget *inside the frame*, NOT the frame budget itself** ŌĆö the 16.67 ms registry `t1_latency_max` constant is a *different* constraint (Event Bus per-channel latency window); AC-P1's end-to-end Pillar 5 Feedback Floor is a *third, separate* clock (100 ms at 60 fps target / 133 ms at 30 fps floor ŌĆö touch-began ŌåÆ render-frame-submission wall-clock). All three must hold independently.
- **R13.d T1 subscriber cap + per-handler cost (independent caps).** No `input.*` channel may have more than **4** T1 subscribers simultaneously registered in a given scene (SubscriberCountMax = 4, locked). Per-handler cost threshold: **1.0 ms Release / 1.5 ms Dev** on the ADR-003 reference device. **Noise-floor arithmetic (pass-5 BLOCKING #9 correction):** the pass-3/4 derivation assumed `Stopwatch.GetTimestamp` 99p tail noise of 0.4ŌĆō0.6 ms on Mali-G52-class ARM would carry through to the Release-tier `ProfilerMarker` path; that assumption was wrong. `ProfilerMarker.Begin/End` bracket cost is sub-microsecond per bracket on ARM Cortex-A53/A55 silicon (per [Unity Profiling Core API guide](https://docs.unity.cn/Packages/com.unity.profiling.core@1.0/manual/profilercounter-guide.html) ŌĆö ProfilerMarker compiles to native intrinsics, not a managed-code Stopwatch call), so Release-path 99p noise is materially lower than the Dev-path Stopwatch 99p noise. The 1.0 ms Release ceiling is therefore not margin-constrained by the Stopwatch-era 0.4ŌĆō0.6 ms figure; it is a design floor that leaves comfortable headroom for compliant handlers (~0.8 ms actual cost headroom against ~0.2 ms residual noise). Dev-path `Stopwatch.GetTimestamp` retains the 0.4ŌĆō0.6 ms 99p tail noise and the 1.5 ms Dev ceiling is sized against that. | **Biome Map** (MVP) | `input.tap.main`, `input.drag.start`, `input.drag.end` | T1 | Both | Level-select tap, swipe-to-pan map |
| **Main Menu / Settings / Pause** (MVP) | `input.tap.main` | T1 | `TapPayload` | Menu navigation |
| **Modal/Dialog + Level End** (MVP) | `input.tap.main` | T1 | `TapPayload` | Modal dismiss, level-end continue |

**Data contracts ŌĆö `InputState.CurrentPointer`:**

```csharp
public readonly struct InputPointerState {
    public Vector2 ScreenPosition;                    // safe-area-local pixels
    public bool IsPressed;                             // true from began to ended/canceled
```

- Platform-specific system events (volume buttons, home gesture, notification tray) ŌĆö handled by Unity's native lifecycle callbacks, not Input
- Keyboard/mouse at MVP ŌĆö no support (touch-only per technical-preferences.md); post-MVP expansion would require an ADR to repeal R2 and add device abstraction
- In-Editor rapid-iteration keyboard shortcuts for playtesters ŌĆö live in `DEVELOPMENT_BUILD`-gated `EditorPlaytestInput.cs`, which IS on Input's allowlist but does not publish on Event Bus (writes directly to playtest log per CD-SYSTEMS ownership rule)

## Formulas

Input System is a dispatch layer with minimal continuous mathematics. Most threshold values (safe-area clipping, suppression durations, occlusion constants) are tuning knobs defined in Section G and referenced by rule in Section C. Two calculations require formal treatment because they have non-trivial boundary conditions or produce outputs that downstream CI tests key on.

### D.1 InputPipelineBudget

**Purpose.** Formalizes the latency constraint from R13 so that CI assertions and per-handler budget decisions have a single symbolic source to reference.

**Expression.**

```
TotalLatency_ms = PreFire_ms + T1Chain_ms

where:
  PreFire_ms   Ōēż BudgetCeiling_ms / 2
  T1Chain_ms   Ōēż BudgetCeiling_ms / 2
  TotalLatency_ms Ōēż BudgetCeiling_ms

  BudgetCeiling_ms = 4.0    if DeviceProfile = Release   (Input's main-thread sub-budget inside a 16.67 ms frame)
  BudgetCeiling_ms = 5.0    if DeviceProfile = Dev       (absorbs profiler-marker overhead)
```

**Variable table.**

| Symbol | Type | Range | Description |
|--------|------|-------|-------------|
| `PreFire_ms` | float | 0 ŌĆō 2.0 (Release) / 0 ŌĆō 2.5 (Dev) | Wall-clock ms from `EnhancedTouch.activeTouches` read through `TapPayload` / `DragPayload` construction to `EventBus.Fire()` entry. Measured inside `InputDispatcher.Update()`. |
| `T1Chain_ms` | float | 0 ŌĆō 2.0 (Release) / 0 ŌĆō 2.5 (Dev) | Wall-clock ms from `EventBus.Fire()` entry through last T1 subscriber return on any `input.*` channel. Each subscriber's contribution is additive and serial. |
| `BudgetCeiling_ms` | float | {4.0, 5.0} | Input's main-thread sub-budget inside a 16.67 ms frame ŌĆö **NOT** the registry `t1_latency_max` (which is Event Bus's per-channel T1 window, a different clock) and **NOT** AC-P1's end-to-end Pillar 5 clock (100 ms at 60 fps target / 133 ms at 30 fps floor ŌĆö touch-began ŌåÆ render-frame-submission). 4.0 = Input's Release claim on the frame (leaves Ōēź12.67 ms for render + physics + other T1 channels). 5.0 = Development Build allowance for profiler-marker overhead (~0.1 ms ├Ś markers + IL2CPP-None vs IL2CPP-High stripping differences). |
| `DeviceProfile` | enum | {Release, Dev} | Build profile at CI time. Release = ADR-003 reference device with IL2CPP High stripping. Dev = editor or development build with profiler instrumentation enabled. |
| `TotalLatency_ms` | float | 0 ŌĆō 5.0 (bounded by ceiling) | Sum of PreFire and T1Chain. Must not exceed BudgetCeiling for the active profile. Output is clamped by CI assertion ŌĆö an overrun fails the test, it does not silently continue. |

**Output range.** Bounded above by `BudgetCeiling_ms` (4.0 Release / 5.0 Dev); bounded below by 0 (theoretical; minimum measured overhead is ~0.4 ms for empty subscriber chain). The ceiling is a CI hard gate, not a runtime clamp.

**Worked example ŌĆö nominal Release build, 3 subscribers on `input.tap.main`.**

```
PreFire_ms   = 1.1   (classifier + two raycasts + payload alloc)
T1Chain_ms   = 0.9   (HUD 0.3ms + Collection 0.4ms + Modal/Dialog 0.2ms)
TotalLatency = 1.1 + 0.9 = 2.0 ms   ŌåÆ   2.0 Ōēż 4.0   PASS

DeviceProfile = Dev, same chain:
BudgetCeiling = 5.0 ms
2.0 Ōēż 5.0   PASS (comfortable margin)

(Tool System does not count here ŌĆö it polls `InputState.CurrentPointer`
 in Update per R3.d rather than subscribing to input.tap.main.)
```

**Rationale.** The half-and-half split (PreFire Ōēż half, T1Chain Ōēż half) is a **design-intent load-balancing contract**, not a merge-blocking gate ŌĆö clarified per pass-4 BLOCKING #8. It prevents one side from crowding out the other at the design level: a slow raycast pass cannot steal budget from subscribers, and a slow subscriber cannot justify loosening the pre-fire cost. The asymmetric Dev ceiling absorbs Stopwatch overhead without changing the Release target ŌĆö the production experience is unchanged.

**Gate asymmetry note (pass-4 BLOCKING #8 clarification)**: only the **joint sum** (AC-R13c Release 99p-of-per-frame-sum, 4.0 ms ceiling) is a BLOCKING CI merge gate. The half-and-half per-component caps (AC-R13a PreFire 99p Ōēż 2.0 ms; AC-R13b T1Chain 99p Ōēż 2.0 ms) are classified PLAYMODE ŌĆö they produce attribution data for failing joint-sum runs but do NOT independently block merge. This is deliberate: a build with `(PreFire = 2.5, T1Chain = 1.4, joint = 3.9)` passes AC-R13c (3.9 Ōēż 4.0) and merges, accepting the sub-budget imbalance as a real-but-tolerable deviation from the design-intent split. A build with `(PreFire = 2.5, T1Chain = 2.0, joint = 4.5)` fails AC-R13c and is merge-blocked (or correlated-tail-downgraded per AC-R13c's pass-4 routing clause). Designers reading D.1 should expect the half-and-half split to be violated occasionally under thermal pressure and read AC-R13a/b failures as attribution signals for AC-R13c triage, not as independent merge blockers.

The 4.0 ms Release ceiling is explicitly a *sub-budget* inside one 16.67 ms frame ŌĆö prior revisions of this GDD conflated Input's main-thread cost with the full frame budget (same number, three different clocks); that conflation is resolved here per the 2026-04-22 review.

**Validation.** CI PlayMode test on ADR-003 reference device: 1000 synthetic touch events across 60 frames; 99th-percentile `TotalLatency_ms Ōēż 4.0`; mean `PreFire_ms < 1.5` and mean `T1Chain_ms < 1.5`. Dev build gate: 99th-percentile `TotalLatency_ms Ōēż 5.0`.

### D.3 GestureClassification

**Purpose.** Defines the piecewise mapping from raw touch input `(d, t)` to gesture class. Replaces an implicit decision tree with explicit, gap-free boundary conditions so implementers and reviewers can verify completeness and testers can construct boundary-value cases mechanically.

**Expression.**

Let `d` = cumulative movement distance in platform-independent units (pt on iOS, dp on Android), and `t` = touch duration in milliseconds from `began` phase to current frame. `v` = finger lift velocity at `ended` (magnitude of last-frame delta / last-frame delta-time).

```
GestureClass(d, t, ended, v) =

  # Branch priority (pass-5 BLOCKING #4 resolution ŌĆö explicit precedence):
  # rules are evaluated top-to-bottom; first-match wins.
  # Drag branches precede InProgress branches because Drag outputs are
  # classifier-terminating (publish drag.start and lock via D.3 Statefulness),
  # whereas InProgress outputs are transient (no publish). At any (d, t, ended, v)
  # combination where both a Drag rule AND an InProgress rule would match,
  # Drag wins ŌĆö per this precedence, NOT a separate tiebreaker.

  Drag         if  d Ōēź d_slop                           # immediate drag ŌĆö threshold crossing, any t
  Drag         if  d Ōēź d_early  AND  t Ōēź t_early        # early-drag ŌĆö slow-deliberate-drag catch

  Tap          if  ended  AND  d < d_slop  AND  t Ōēż t_tap_max        # standard tap
  Tap          if  ended  AND  d < d_slop  AND  t > t_tap_max        # late-lift tap forgiveness
               AND  v < v_lateLift                                    # (motor-impaired accommodation, R3.a)

  LongPress    if  ended  AND  d < d_slop  AND  t > t_tap_max        # published as
               AND  v Ōēź v_lateLift                                    # input.gesture.longpress.unhandled
                                                                      # on T2 per R3.c; HUD REQUIRED MVP
                                                                      # subscriber per AC-R3d-required

  InProgress   otherwise                                # DEFAULT ŌĆö awaiting classification or lift
                                                        # (pass-5 BLOCKING #4 coverage-hole fix ŌĆö
                                                        #  subsumes pass-4 rules 1 & 2 AND covers
                                                        #  the (d < d_early, t > t_tap_max, ended=false)
                                                        #  case that previously matched no branch)
```

**Boundary semantics.** At `d == d_slop` the Drag branch fires ŌĆö the `Ōēź` comparator is inclusive; the Tap branch's `d < d_slop` is strict. Similarly, `v == v_lateLift` at `ended` classifies as LongPress (not Tap) because the Tap late-lift branch is `v < v_lateLift` strictly. The `t Ōēż t_tap_max` boundary is inclusive ŌĆö a touch ending at exactly `t == t_tap_max` with `d < d_slop` classifies as Tap (standard branch, not late-lift). These boundary rules are load-bearing for H.2 Formula Correctness ACs ŌĆö implementers and QA should read the comparators exactly as written.

**Priority and coverage-hole invariants (pass-5 BLOCKING #4 resolution).** Two structural defects in the pass-4 rule block are fixed in the revision above:

1. **Priority ambiguity at Path B canonical case.** At `(d = 7, t = 160 ms, ended = false)` under defaults `(d_slop = 10, d_early = 6, t_early = 150 ms, t_tap_max = 300 ms)`, pass-4's rule ordering had `InProgress` rule 1 (`t Ōēż 300 AND d < 10 AND NOT ended` ŌåÆ MATCH) AND `Drag` early-branch (`d Ōēź 6 AND t Ōēź 150` ŌåÆ MATCH) both firing with no stated precedence. The revised top-to-bottom evaluation order places Drag branches first, so Path B canonical case unambiguously produces `Drag` (publish `drag.start` this frame per the worked example). Implementers MUST evaluate in the written order.
2. **Coverage hole at `d < d_early, t > t_tap_max, ended = false`.** Under pass-4's five explicit rules (InProgress ├Ś2 + Tap ├Ś2 + Drag ├Ś2 + LongPress), a gesture with `d = 3, t = 320 ms, ended = false, d_slop = 10, d_early = 6, t_tap_max = 300 ms` matched no branch ŌĆö InProgress rule 1 failed (t > 300), InProgress rule 2 failed (t > t_early implicitly), Tap rules failed (ended = false), Drag rules failed (d < d_early), LongPress failed (ended = false). The classifier would silently fall through in implementation. The revised `InProgress otherwise` default rule covers this ŌĆö the gesture remains InProgress awaiting lift, then classifies on the `ended` frame via Tap / LongPress rules. Deterministic fixture: `(d = 3, t = 320 ms, ended = false)` ŌåÆ `InProgress`; same gesture at `(d = 3, t = 350 ms, ended = true, v = 0.05 pt/ms)` ŌåÆ `Tap` (late-lift forgiveness branch with default `v_lateLift = 0.1 pt/ms`).

**Statefulness.** The classifier is **stateful per-gesture**. Once a gesture publishes `input.drag.start` (the Drag branch fires on `d Ōēź d_slop` OR on the early-drag gate `d Ōēź d_early AND t Ōēź t_early`), the gesture is locked into Drag for the remainder of its lifetime ŌĆö subsequent frames cannot re-classify it as Tap or LongPress even if `d` somehow decreases or other conditions would otherwise match. The Drag lock persists until `ended` or `canceled`, at which point the classifier publishes `input.drag.end` and releases. This prevents pathological re-classification under finger-jitter or floating-point noise (a player whose drag slightly retracts mid-gesture does not see their drag disappear and become a tap). Implementers MUST maintain an explicit `_activeClassification` field on the gesture tracker and short-circuit the classifier whenever it holds `Dragging`.

**Classifier-knob margin invariant.** The OnValidate rule on the classifier ScriptableObject MUST enforce `d_early Ōēż d_slop ŌłÆ 3` (a 3-unit margin, in pt/dp). The prior rule `d_early < d_slop` permitted settings like `(d_slop = 20, d_early = 19)` which produce a classifier **dead zone** at `d = 15, t = 160 ms, ended = false`: the touch satisfies neither Tap rules (`d < d_slop` requires `ended`, which is false), nor the early-drag gate (`d = 15 < d_early = 19`), nor the immediate-drag gate (`d = 15 < d_slop = 20`), so it remains `InProgress` indefinitely until lift ŌĆö burning the 100 ms Pillar 5 response window at the accessibility ceiling tuning. The 3-unit margin ensures `d_early Ōēż 17` when `d_slop = 20`, so a deliberate slow drag at `d = 15` still classifies as `InProgress` approaching the `d_early` gate, and a drag at `d = 17` classifies as Drag via the early-drag branch at `t Ōēź t_early`. Reviewers should reject any PR raising `d_slop` without confirming the margin holds. See AC-D3-margin for the validation path.

**Variable table.**

| Symbol | Type | Range | Description |
|--------|------|-------|-------------|
| `d` | float | 0 ŌĆō unbounded (pt/dp) | Cumulative movement distance since `began`. Computed as sum of per-frame deltas, not straight-line origin distance (handles circular motion correctly). |
| `t` | float | 0 ŌĆō unbounded (ms) | Duration since the touch entered `began` phase on the current classifier frame. |
| `d_slop` | float | 10 ŌĆō 20 (pt/dp) | Tap movement tolerance. Default 10. Accessible range 10ŌĆō20 (Section G). Below 10: unintended drag promotions from normal motor jitter. Above 20: intended drags misclassified as taps on short swipes. |
| `d_early` | float | 6 ŌĆō (d_slop ŌłÆ 3) (pt/dp) | Early-drag trigger distance used only when `t Ōēź t_early`. Default 6. Invariant: `d_early Ōēż d_slop ŌłÆ 3` enforced by `OnValidate` (3-unit margin prevents a classifier dead zone under max-accessibility tuning ŌĆö see D.3 Classifier-knob margin invariant). |
| `t_tap_max` | int | 300 ŌĆō 600 (ms) | Tap duration ceiling. Default 300 ms. Accessible range 300ŌĆō600 ms (Section G). |
| `t_early` | int | 100 ŌĆō 200 (ms) | Early-drag trigger time. Default 150 ms. At `t Ōēź t_early`, the smaller `d_early` threshold activates to catch deliberate slow Rush-phase drags before the full tap window closes. |
| `v` | float | 0 ŌĆō unbounded (pt/ms) | Finger lift velocity at `ended` phase. Used only for late-lift forgiveness branch. |
| `v_lateLift` | float | 0 ŌĆō 0.8 (pt/ms) | Velocity ceiling for late-lift Tap forgiveness. **Default 0.1 pt/ms (pass-5 BLOCKING #7 recalibration from 0.02).** Touches at or above this velocity at lift (with `t > t_tap_max` and `d < d_slop`) classify as LongPress and publish `input.gesture.longpress.unhandled` on T2 (HUD REQUIRED MVP subscriber dual-channel); they are NOT discarded. Touches below this velocity fall through to Tap late-lift forgiveness. **Derivation**: pass-5 systems-designer BLOCKING #7 reported real motor-impaired lift velocities 0.05ŌĆō0.8 pt/ms (consistent with [touch-motor-impairment kinematic-theory research](https://www.researchgate.net/publication/324659828_Gesture_Input_for_Users_with_Motor_Impairments_on_Touchscreens_Empirical_Results_based_on_the_Kinematic_Theory) and related CHI touchscreen accessibility work); pass-4's 0.02 default sat below the floor of that range, so late-lift Tap-forgiveness was functionally dead for the accommodation population. 0.1 sits above the 0.05 floor (activates for most motor-impaired lifts) but well below the 0.8 upper end (preserves LongPress discrimination for very-fast lifts). Empirical calibration on ADR-003 is tracked in OQ-11. |
| `GestureClass` | enum | {InProgress, Tap, Drag, LongPress} | Output classification. `InProgress` is transient (never published). `Tap` ŌåÆ `input.tap.main` (T1). `Drag` ŌåÆ `input.drag.start` on threshold crossing (T1). `LongPress` ŌåÆ `input.gesture.longpress.unhandled` (T2 Feedback; **HUD is REQUIRED MVP subscriber per R3.c + AC-R3d-required**, dual-channel: offset visual + audio cue; post-MVP systems may subscribe alongside HUD). |

**Output range.** Categorical, four values. Every `(d, t, ended, v)` combination maps to exactly one class ŌĆö the piecewise definition is exhaustive and mutually exclusive. `InProgress` is never published to Event Bus; it is an internal state. The late-lift forgiveness branch and the `LongPress` branch are the only cases that require the `ended` signal to resolve ŌĆö all `Drag` promotions fire on the frame the threshold is crossed, not on lift.

**Worked examples ŌĆö three paths.**

```
Path A ŌĆö clean tap:
  d=3pt, t=180ms, ended=true, v=0.01 pt/ms
  d < d_slop(10), t Ōēż t_tap_max(300), v < v_lateLift(0.02)   ŌåÆ   Tap
  Publish input.tap.main

Path B ŌĆö slow deliberate drag (Rush-phase ramp placement):
  d=7pt, t=160ms, ended=false
  d Ōēź d_early(6) AND t Ōēź t_early(150)   ŌåÆ   Drag
  Publish input.drag.start this frame
  (without early trigger: d=7 < d_slop(10); touch would remain InProgress until t=300ms ŌĆö 140ms late)

Path C ŌĆö long press (fast lift after long hold):
  d=2pt, t=420ms, ended=true, v=0.2 pt/ms
  d < d_slop, t > t_tap_max(300), v Ōēź v_lateLift(0.1)   ŌåÆ   LongPress
  Publish input.gesture.longpress.unhandled on T2 Feedback channel
  (HUD REQUIRED MVP subscriber per R3.c renders dual-channel feedback:
   offset visual tick above thumb + audio tick via Audio Bus)

Path D ŌĆö late-lift Tap forgiveness (motor-impaired, slow lift ŌĆö pass-5 BLOCKING #7 default):
  d=2pt, t=420ms, ended=true, v=0.05 pt/ms
  d < d_slop, t > t_tap_max(300), v < v_lateLift(0.1)   ŌåÆ   Tap
  Publish input.tap.main (late-lift Tap forgiveness now active at default ŌĆö
  pass-4 default 0.02 pt/ms left this branch functionally dead; pass-5
  recalibration to 0.1 pt/ms aligns with motor-impaired lift velocity
  range 0.05ŌĆō0.8 pt/ms per D.3 variable table derivation.)
```

**Rationale.** The early-drag trigger (Path B) is the load-bearing case for Rush-phase usability: without it, a deliberate 150 ms slow drag sits in `InProgress` until the 300 ms tap window closes, wasting half of the Tool System's 100 ms response window. The two-variable threshold `(d_early, t_early)` lets tuning tighten or loosen the slow-drag sensitivity without touching the standard slop, keeping tap and drag classifier sensitivities independent. The piecewise form makes boundary tests mechanically derivable ŌĆö every variable has an exact threshold, so QA writes `(threshold ŌłÆ 1, at-threshold, threshold + 1)` cases without guessing.

**Validation.** Unit tests covering all eight boundary conditions: `d = d_slop ŌłÆ 1`, `d = d_slop`, `d = d_slop + 1`, `t = t_tap_max`, `t = t_early ŌłÆ 1`, `t = t_early` (with `d = d_early`), late-lift forgiveness path, long-press discard path. Integration test: simulated slow Rush-phase drag (`d = 7pt` at `t = 160ms`) confirms `input.drag.start` fires at frame Ōēł 160 ms, not frame 300 ms.

### Formula Candidates Moved to Tuning Knobs

The following geometric and capacity constants are tuning knobs (Section G), not formulas: `occlusion_offset_dp` (default 72) and `edge_padding_dp` (default 8) from R7's spawn-above-finger clamp `min(touch.y + offset, safeArea.top ŌłÆ padding)`; `PhaseTransitionSuppressDuration_ms` from R10.a; `SubscriberCountMax` (4) and `PerHandlerCostThreshold_ms` (1.0 Release / 1.5 Dev per pass-3 retune, independent of SubscriberCountMax per R13.d) from R13.d. Each is a constant with a tuning range ŌĆö no interacting variables, no emergent output. Treating them as formulas would be padding.

## Edge Cases

### OS / Platform

- **If `_activeTouchId` is tracked but the ID vanishes from `EnhancedTouch.Touch.activeTouches` without firing `ended` or `canceled`** (OS-level touch steal ŌĆö Android OEM gesture navigation, notification tray swipe, Samsung edge panel): `InputDispatcher.Update()` performs an absence-detection pass ŌĆö if the tracked ID is not present in `activeTouches` AND no `ended`/`canceled` was received, treat as an implicit cancel: clear `_activeTouchId`, abandon the in-flight gesture, publish nothing. Log `[InputDispatcher] Tracked touch {id} vanished ŌĆö implicit cancel`. This is independent of R11's pause cancellation, which only covers `OnApplicationPause`.

- **If `ended` or `canceled` arrives for a finger ID that does not match `_activeTouchId`** (spurious OS ghost event; also the aftermath of an implicit cancel): discard silently. No event published, no state change, no log in release builds. Rationale ŌĆö these events are noise at the current abstraction; matching IDs is the only authoritative signal that a tracked gesture ended.

- **If `Screen.safeArea` changes mid-session** (Android foldable unfold, split-screen activation): out of MVP scope. Money Miner is portrait-locked per `technical-preferences.md` and foldables are not a supported form factor for v1.0. If a foldable unfolds during play, the cached safe-area rect becomes stale and edge-adjacent touches may misbehave until scene reload. Post-launch consideration: add a `Screen.safeArea` change check in `Update` and re-cache + emit a `DeviceDisplayMetrics` changed event.

- **If the OS recycles a touch ID after a tracked finger vanished implicitly** (EC-OS1 fired, new touch arrives with the same finger ID the OS just recycled): a `began` matching `_activeTouchId` while `IsPressed` is already true (compound failure: EC-OS1's absence-detection did not run yet) is treated as an implicit cancel of the prior gesture before starting fresh ŌĆö NOT silently skipped. Without this rule, the prior gesture's state would leak into the new touch. Enforcement ŌĆö PlayMode test simulating OS ID recycling.

### State Machine / Lifecycle

- **If `SetLoadSuppression(true)` fires mid-drag**: R10.c abandons the in-flight gesture and clears `_activeTouchId`. When load suppression lifts and the Unity Input System delivers the physical `ended` event for the touch that was still down during load, the arriving `ended`'s finger ID does NOT match `_activeTouchId` (cleared during abandonment). The ghost `ended` is discarded under the EC-OS2 rule ŌĆö no event published. This is why EC-OS2's ID-mismatch rule is load-bearing for suppression recovery.

- **If `PushModalSuppression(modalId)` is called during a T1 subscriber handler that was triggered by Input's own `Fire`** (Modal/Dialog opens in response to a tap, calling into Input's suppression API while Input is still inside its dispatch stack): `PushModalSuppression` sets the suppression flag atomically but gesture abandonment is DEFERRED to the next `InputDispatcher.Update()` entry. Abandonment code does NOT run from inside `PushModalSuppression` ŌĆö re-entering the state machine from an arbitrary subscriber call stack depth would corrupt state.

- **If `InputDispatcher.Initialize()` is called before `EnhancedTouchSupport.Enable()` completes, or called a second time without an intervening `Dispose()`** (scene reload, additive scene load): `Initialize()` asserts `State == Uninitialized` at entry and throws `InvalidOperationException` with message `[InputDispatcher] Already initialized ŌĆö call Dispose() before re-initializing`. Double-registration of the `gameplay.level.phase_changed` subscription would cause every phase change to trigger two suppression entries. Enforcement ŌĆö integration test.

- **If a T1 subscriber mutates Input's suppression state inside its own handler** (e.g., Tool System places a tool ŌåÆ fires `gameplay.level.quota_progress` ŌåÆ Level Runtime emits `gameplay.level.phase_changed` ŌåÆ Input's subscription wants to enter `Suppressed.PhaseTransition`): Input snapshots suppression flags at `Update()` entry and uses the snapshot for the entire frame's processing. Suppression mutations during the T1 chain take effect the FOLLOWING frame, not mid-dispatch. Prevents re-entrancy hazards in the dispatcher.

### Classifier Boundary

- **If a frame drops to Ōēź 200 ms (severe thermal throttle or background work stall)**: D.3 uses `t = elapsed since began`, sampled per `Update`. If `began` arrives at frame N, frame N takes 220 ms, finger lifts mid-frame ŌĆö `t` at `ended` is 220 ms Ōēż `t_tap_max(300)`, fires as Tap correctly. **Dangerous case**: if `began` is processed in frame N (220 ms duration) and finger lifts at the START of frame N+1 (also 220 ms), `t` at `ended` = 440 ms with near-zero `v` ŌåÆ classified as LongPress. Per revised R3.c, this publishes `input.gesture.longpress.unhandled` on T2 (not silently discarded); the player's intended quick tap still does not reach Tap subscribers, but the T2 event exists so opt-in feedback consumers can surface "hold received" UI. A player-intended quick Tap-subscription semantic is still lost to frame-rate collapse. Accepted for MVP ŌĆö Pillar 5 assumes a healthy 60 fps / 30 fps floor; sustained sub-5 fps is a thermal emergency, not a classification concern. Flag in Section I (Open Questions) if post-MVP telemetry shows late-lift taps disproportionately lost on specific devices.

- **If a second finger's `began` and the first finger's `ended` arrive in the same `Update`** (rapid multi-touch): Input processes the tracked finger's lifecycle event FIRST (publish tap / end drag, clear `_activeTouchId`), THEN evaluates any new `began`. If the new `began` is from a second finger, it is ignored per R2 (first-touch only). Processing in the wrong order would leave the first finger's gesture dangling and the second finger's `began` silently consumed. Enforcement ŌĆö unit test with synthetic activeTouches buffer containing both phases.

- **If `t_tap_max` is raised to the accessibility ceiling (600 ms)** and a player holds a stationary finger for 580 ms with a slow lift (`v < v_lateLift`): classified as Tap per D.3 late-lift forgiveness. **UX discontinuity to document for playtesters**: under the 600 ms setting, extended stationary holds the player intended as "pause to think" fire as taps. This is a cost of motor-impaired accommodation. Flag in Section I for VS Accessibility Service UX review ŌĆö the VS settings UI should describe this trade-off in the accessibility menu copy.

### Hit-Test Resolution

- **If the hit-tested GameObject (UI or world) is destroyed between the raycast and `EventBus.Fire`**: after `OverlapPointNonAlloc` or `RaycastAll`, Input performs a null-and-destroyed check (`target == null || !target`) before populating `TapPayload.Target`. If destroyed, `Target = null` (treat as playfield tap). Failing this check would propagate a phantom-object reference to up to 4 T1 subscribers (R13.d cap), causing cascading `MissingReferenceException`. Enforcement ŌĆö PlayMode test destroying the target object synchronously inside an earlier handler.

- **If multiple UI canvases share the same `sortingOrder` at the tap point** (authoring error): `EventSystem.RaycastAll` returns results in sort-then-distance order; at identical sort order the result order is Unity-implementation-defined. A `DEVELOPMENT_BUILD` scene-validator asserts every active canvas in the scene has a unique sort order. Runtime recovery is NOT attempted ŌĆö Input cannot guess authoring intent. Release builds accept whatever Unity returns; the scene validator in Dev catches this before release.

- **If the tap falls on a world collider whose layer is excluded from Input's layer-filtered mask**: `OverlapPointNonAlloc` returns zero hits; `Target = null`; the player perceives the tap as "missing" a visible object. This is intentional ŌĆö Input's layer mask defines which world objects are tap-eligible, and excluded objects are visual-only per design (Data Registry + Tool System decide which ToolConfigs register on tap-eligible layers). If a consuming system needs expanded hit eligibility, it updates its collider's layer, not Input's mask. Documentation handoff to consumer GDDs.

- **If a touch begins at exactly the `Screen.safeArea` boundary pixel**: inclusive of the safe area. The rule is `touch.position IN safeArea` (closed interval on all four edges). Only touches strictly OUTSIDE the safe-area rectangle are suppressed. Edge-of-screen taps are a common player affordance (reaching a button with the edge of the thumb) and must not be lost to a strict-inequality bug.

### Configuration / Tuning Extremes

- **If a consuming system queries `InputState.DeviceDisplayMetrics` before `InputDispatcher.Initialize()` completes** (Unity script execution order violation): the query throws `InvalidOperationException` with message `[InputDispatcher] DeviceDisplayMetrics not populated ŌĆö InputDispatcher must execute before consumers in Script Execution Order`. Silent return of a default value would mask authoring errors in the scene setup. `InputDispatcher` is pinned to a negative Script Execution Order number (e.g., -1000) so it always initializes before default-order MonoBehaviours. Enforcement ŌĆö integration test that instantiates a consumer MonoBehaviour at default order and asserts the InvalidOperationException fires if Input is mispinned.

### Cross-System Ownership Rule

**Input System owns:**
- Touch capture and the sole `UnityEngine.InputSystem.*` access boundary (R1)
- Gesture classification thresholds ŌĆö tap, drag, long-press boundary rules (R3, formula D.3)
- Safe-area coordinate contract ŌĆö all published positions in safe-area-local pixels (R5)
- Finger-occlusion offset constants (R7)
- `DeviceDisplayMetrics` read-only struct (R6.a)
- Suppression gate ŌĆö state, priority, and transition (R10)
- Pause-gesture cancellation (R11)
- Input pipeline budget enforcement (R13, formula D.1)

**Consumers own:**
- **Hit-target authoring** ŌĆö collider and RectTransform bounds, 1.4├Ś visual-sprite inflation, 44pt iOS / 48dp Android accessibility floor (R6 ŌĆö Input does NOT enforce; R6.b debug-build floor assertion is a safety net, not a replacement for consumer discipline)
- **Feedback emission** ŌĆö every sound, visual, animation, haptic, and particle is subscriber-side (R8 ŌĆö Input produces zero)
- **Gameplay interpretation** ŌĆö whether a tap hits a coin, a drag places a ramp, a swipe advances a map (R12 ŌĆö Input does NOT route by gameplay class)
- **Per-handler budget** ŌĆö each T1 subscriber keeps work Ōēż 2ms on reference device or defers via `Task.Run` / next-frame `EventBusPlayerLoopSystem` (R13.d)
- **Multi-touch semantics (post-MVP)** ŌĆö if a consumer needs pinch-zoom or dual-thumb gestures, it repeals R2 via ADR; Input does not silently widen its contract

**Three scope guardrails:**
- **Settings UI is not Input's.** If a value is player-facing (keybinding, tap-sensitivity slider), it belongs in Accessibility Service. Input reads accessibility state via `PlayerPrefs` (R9); Input does not own the settings screen.
- **Gameplay rules are not Input's.** Hit priority, target eligibility, and collision-layer assignment belong in the consumer (Tool, Collection, Modal). Input reports contact; consumers decide meaning.
- **Platform differences are not consumers'.** iOS vs Android differences (pt vs dp, safe-area shape, notch geometry) live inside `DeviceDisplayMetrics` (R6.a). Consumers do not `#ifdef` ŌĆö they query the struct.

If these guardrails are violated, the consumer is encoding Input's contract in two places and one will eventually drift.

## Tuning Knobs

Input System's tuning knobs are a mix of **perceptual thresholds** (gesture classifier ŌĆö affects feel and accessibility), **geometric constants** (finger-occlusion offset, safe-area padding ŌĆö affects ergonomic correctness on each aspect ratio), **pipeline capacity** (subscriber cap, per-handler threshold ŌĆö affects the 16.67 ms T1 budget), and **CI severities** (R1/R12 grep gates ŌĆö affects architectural discipline). Gameplay tuning (hit priority, collision layers, target eligibility) lives in the consumer GDDs per R12, not here.

The R9 `AccessibilitySettings` struct's three values (`t_tap_max`, `d_slop`, long-press threshold) are exposed at MVP via `PlayerPrefs` with VS Accessibility Service UI as their designer-facing surface. ### Interaction Between Tuning Knobs

- **Coupled ŌĆö `d_slop` and `d_early`**: the formula D.3 requires `d_early Ōēż d_slop ŌłÆ 3` (3-unit margin; see OnValidate rule and D.3 Classifier-knob margin invariant). Tuning `d_slop` upward (accessibility) without also tuning `d_early` produces a larger dead zone where slow drags are neither caught by the early trigger nor by the standard threshold. Change both together, or the slow-drag-to-ramp path (Path B in D.3 worked examples) regresses ŌĆö the margin is the minimum guard against this; aggressive accessibility profiles may want a wider gap.
- **Coupled ŌĆö `t_tap_max` and `v_lateLift`**: at the accessibility ceiling (`t_tap_max = 600 ms`), `v_lateLift` becomes the sole discriminator between late-lift Tap and LongPress for slow finger lifts. Raising `v_lateLift` too aggressively at 600 ms converts nearly every slow lift into a tap, eliminating the LongPress reserve that post-MVP systems may want to claim. Review together when adjusting accessibility profile.
- **Independent caps ŌĆö `SubscriberCountMax` and `PerHandlerCostThreshold_ms`**: prior revisions treated these as multiplicatively coupled (`N ├Ś per_handler Ōēż chain` at `4 ├Ś 0.5 = 2.0 ms`, binding, zero slack). Reversed per review log 2026-04-22 pass-2 BLOCKING #3 ŌĆö the 0.5 ms per-handler threshold sat below `Stopwatch.GetTimestamp` mobile noise (~0.2 ms mean ŌåÆ 2.5:1 SNR, 5ŌĆō8% false-positive CI runs). Pass-2 relaxation raised per-handler Release to 0.8 ms / Dev to 1.2 ms on a mean-based 4:1 SNR claim. **Pass-3 BLOCKING #8 retune**: the mean-based SNR claim was wrong at the tail ŌĆö Mali-G52-class 99p Stopwatch noise is ~0.4ŌĆō0.6 ms, so 0.8 ms tail SNR was still Ōēż 2:1. Pass-3 restores a Ōēź 2:1 tail SNR by raising per-handler Release to **1.0 ms** and Dev to **1.5 ms**, with Release measurement migrated from inline `Stopwatch.GetTimestamp` brackets (~0.1ŌĆō0.2 ms per bracket, 0.4ŌĆō0.8 ms per chain) to inline `ProfilerMarker.Begin/End` brackets (~0.05ŌĆō0.1 ms per bracket, 0.2ŌĆō0.4 ms per chain) + external `ProfilerRecorder` sample reads (pass-4 BLOCKING #5 correction ŌĆö Release retains inline brackets but they are lighter ProfilerMarker brackets, not Stopwatch; `ProfilerRecorder` alone cannot produce per-handler granularity without markers to read). Under the current values, `4 ├Ś 1.0 = 4.0 ms` per-handler ├Ś subscriber count mathematically exceeds the 2.0 ms chain ceiling ŌĆö that is the expected geometry when the two caps are independent: the chain ceiling (AC-R13b) is the authoritative joint budget; the per-handler cap (AC-R13f) is a per-individual fairness rule. In practice, most subscribers sit well below 1.0 ms; the per-handler cap catches single-subscriber regressions the chain-sum average would otherwise hide. Chain-sum failures (> 2.0 ms 99p) and per-handler failures (any one handler > 1.0 ms Release at 99p) are independently reported CI failures. Each knob's legal range is governed by its own row in the Pipeline Capacity Knobs table; there is no cross-knob invariant to preserve. The pre-2026-04-22-pass-2 binding-invariant framing is explicitly repealed. **Dev/Release CI profile selection (pass-3 BLOCKING #3 resolution + pass-4 BLOCKING #4 injection-mechanism spec)**: the `MONEYMINER_RELEASE_GATE` scripting-define-symbol governs which AC tier is active on a given CI build ŌĆö BLOCKING CI Release gates (AC-R13c Release sub-budget, AC-R13f Release per-handler, AC-D1-release, AC-P2) compile-in only under that define via `#if MONEYMINER_RELEASE_GATE` lexical guards (NOT `[Conditional("MONEYMINER_RELEASE_GATE")]` ŌĆö the attribute strips call sites not method bodies and interacts poorly with IL2CPP High stripping; `#if` is the authoritative pattern); Dev PLAYMODE tiers (AC-R13d, AC-R13f Dev, AC-D1-dev) compile-in under `#if UNITY_EDITOR || DEVELOPMENT_BUILD`. The CI pipeline sets the define per build variant (Release-artifact build vs. Development-build) so a Dev build cannot silently satisfy a Release BLOCKING gate.

**CI injection mechanism (pass-5 BLOCKING #10 resolution ŌĆö pass-4 spec rewritten with Unity 6.3 LTS doc citations)**: Unity 6.3 LTS does NOT accept `-define` as a command-line argument to the test runner or builder. The define must be injected via Unity editor scripting invoked from CI. Pass-4's specified mechanism was broken on two fronts per pass-5 performance-analyst + unity-specialist convergence; both are corrected below.

**Pass-4 defect #1 ŌĆö CLI flag conflict.** Pass-4 proposed `game-ci/unity-test-runner@v4 customParameters: -executeMethod MoneyMiner.CI.ReleaseGateSetter.EnableReleaseGate` as a pre-build step on the test-runner action invocation. This is incorrect: the [game-ci/unity-test-runner action](https://game.ci/docs/github/test-runner/) invokes Unity with `-runTests` under the hood; Unity's CLI treats `-runTests` and `-executeMethod` as **mutually exclusive** in a single Unity process invocation ŌĆö when `-runTests` is present, Unity runs the Test Framework and does not execute the supplied `-executeMethod`. `customParameters` on the test-runner action passes additional flags to Unity's test invocation, not a pre-build editor-script hook. The correct pattern requires a **separate step** before the test-runner step:

```yaml
# CI workflow excerpt (Release-artifact jobs only; Dev jobs omit this step)
- name: Inject MONEYMINER_RELEASE_GATE define
  uses: game-ci/unity-builder@v4
  with:
    unityVersion: 6000.3.f1         # Unity 6.3 LTS
    targetPlatform: Android         # (and a parallel iOS step)
    customParameters: -executeMethod MoneyMiner.CI.ReleaseGateSetter.EnableReleaseGate -quit -batchmode -nographics
    buildName: ReleaseGateInjection
    # unity-builder invokes Unity with -executeMethod; no -runTests here

- name: Run Release PlayMode tests
  uses: game-ci/unity-test-runner@v4
  with:
    unityVersion: 6000.3.f1
    testMode: playmode
    # customParameters deliberately omitted ŌĆö test-runner runs -runTests which
    # reads the persisted define set by the prior step's editor-script call
```

The two steps share the project's `ProjectSettings/` state (the scripting-define symbol persists in `ProjectSettings/ProjectSettings.asset`, which is written by `AssetDatabase.SaveAssets()` after `PlayerSettings.SetScriptingDefineSymbols(...)` ŌĆö pass-6 Phase 6D unity-specialist RECOMMENDED prose correction; pass-5 Phase 6A's "`Library/` + `ProjectSettings/`" was slightly imprecise on where the define actually lives). Default game-ci persistence via the runner's cached project checkout carries `ProjectSettings/` across the two steps, so the define set by the builder step persists into the test-runner step. A complementary `DisableReleaseGate()` invocation via a third `unity-builder@v4` teardown step at job-end prevents define-set persistence across CI runs (or alternatively, a workflow-level `actions/cache` scope that discards mutations after the job ŌĆö implementation detail owned by `/test-setup` skill and devops-engineer).

**Pass-4 defect #2 ŌĆö Unity 6.3 API signature.** Pass-4 specified `PlayerSettings.SetScriptingDefineSymbols(NamedBuildTarget.Android, new[] { "MONEYMINER_RELEASE_GATE", ŌĆ”existingŌĆ” })` with `ŌĆ”existingŌĆ”` as an unspecified placeholder. Unity 6.3 LTS's `PlayerSettings.SetScriptingDefineSymbols` has two verified overloads per [Unity Scripting API reference](https://docs.unity3d.com/ScriptReference/PlayerSettings.SetScriptingDefineSymbols.html): `SetScriptingDefineSymbols(NamedBuildTarget buildTarget, string defines)` (semicolon-delimited) and `SetScriptingDefineSymbols(NamedBuildTarget buildTarget, string[] defines)` (array). Both exist ŌĆö pass-5 unity-specialist BLOCKING #2's claim that the `string[]` overload does not exist is superseded by the Unity 6.3 API reference; the real defect is that pass-4's placeholder `ŌĆ”existingŌĆ”` was not a read-modify-write operation. Without reading existing defines first, the call overwrites all prior defines (wiping any platform-set symbols the project needs). Corrected `SetReleaseGate.cs` spec using the semicolon string overload per [Unity Manual Custom scripting symbols](https://docs.unity3d.com/6000.3/Documentation/Manual/custom-scripting-symbols.html):

```csharp
// tools/ci/SetReleaseGate.cs (to be authored during Technical Setup sprint)
using UnityEditor;
using UnityEditor.Build;

namespace MoneyMiner.CI {
    public static class ReleaseGateSetter {
        const string RELEASE_GATE = "MONEYMINER_RELEASE_GATE";

        public static void EnableReleaseGate() {
            ApplyDefine(NamedBuildTarget.Android, add: true);
            ApplyDefine(NamedBuildTarget.iOS,     add: true);
        }

        public static void DisableReleaseGate() {
            ApplyDefine(NamedBuildTarget.Android, add: false);
            ApplyDefine(NamedBuildTarget.iOS,     add: false);
        }

        static void ApplyDefine(NamedBuildTarget target, bool add) {
            // Read existing defines (semicolon-delimited string per Unity 6.3 API)
            string existing = PlayerSettings.GetScriptingDefineSymbols(target) ?? "";
            var symbols = new System.Collections.Generic.HashSet<string>(
                existing.Split(new[] {';'}, System.StringSplitOptions.RemoveEmptyEntries));

            if (add) symbols.Add(RELEASE_GATE);
            else     symbols.Remove(RELEASE_GATE);

            string combined = string.Join(";", symbols);
            PlayerSettings.SetScriptingDefineSymbols(target, combined);
            AssetDatabase.SaveAssets();  // persist ProjectSettings.asset change
        }
    }
}
```
